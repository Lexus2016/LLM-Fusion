# llm-fusion — Fusion Proxy

> [English](./README.md) | **Русский** | [Українська](./README.ua.md)
>
> Перевод соответствует релизу v0.1.27. Каноническая версия — [английская](./README.md).

> **Четыре модели спорят над вашим запросом, судья оценивает их ответы, а финальная модель пишет тот, который вы увидите.**

llm-fusion — это небольшой прокси, который вы запускаете на собственной машине перед [Ollama Cloud](https://ollama.com). Ваш инструмент для работы с кодом подключается к нему так же, как к любой обычной модели — тот же API, тот же стриминг, те же вызовы инструментов. Разница спрятана за именем модели: попросите `fusion-coder` — и получите панель из четырёх моделей, отвечающих параллельно, судью, который сравнивает сказанное ими, и синтезатор, который пишет финальный ответ. На сложных вопросах эта панель даёт вам несколько менее коррелированных взглядов от разных родословных моделей вместо одного, беспристрастного судью, который разбирает их разногласия и помечает хрупкие или галлюцинированные утверждения, и синтезатор, который сводит их в один ответ, — плюс failover внутри провайдера и защиту полноты, которые не дают одной застопорившейся или упёршейся в лимит модели оборвать ход.

Гонять панель на каждый запрос было бы дорого, поэтому есть ещё `fusion-agents`: быстрый маршрутизатор смотрит на каждый запрос и решает, заслуживает ли он полной панели или хватит одного дешёвого вызова. Чтение файла получает один вызов. Восстановление после заваленных тестов — всю панель.

Без базы данных, без шага сборки, без учётных записей. Node 24, один конфигурационный файл YAML и ваш ключ Ollama Cloud. Прокси говорит на обоих диалектах — OpenAI Chat Completions API и Anthropic Messages API, — так что OpenCode, Claude Code, Continue, Cline, Aider или ваш собственный агентный цикл работают без изменений.

Знакомы с **OpenRouter Fusion**? Та же идея — *думают многие модели, отвечает одна*, — но на вашей машине, прозрачно и с расчётом на долгие агентные циклы. Полное сравнение [ниже](#llm-fusion-против-openrouter-fusion).

## Быстрый старт — 5 минут

Нужны две вещи: **Node.js 24 или новее** и **ключ API [Ollama Cloud](https://ollama.com)**.

```bash
# 1. Заберите код
git clone https://github.com/Lexus2016/LLM-Fusion.git llm-fusion
cd llm-fusion

# 2. Установите зависимости (шага сборки нет — TypeScript исполняется напрямую через tsx)
npm install

# 3. Положите свой ключ Ollama Cloud в .env
printf 'OLLAMA_API_KEY=ollama-your-key\n' > .env

# 4. Запустите прокси
npm start
```

Готово — прокси слушает на `http://127.0.0.1:8080`. Проверьте, что работает:

```bash
curl http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-coder","messages":[{"role":"user","content":"Say hello"}]}'
```

### Шаг 5 — выберите модель под задачу

Направьте любой OpenAI-совместимый клиент на `http://127.0.0.1:8080/v1` (ключ API — любая непустая строка, например `local-no-auth`) и попросите один из трёх готовых пресетов:

| Просите | Когда вы… |
|---|---|
| **`fusion-coder`** | пишете код, планируете функциональность, проверяете изменения |
| **`fusion-researcher`** | исследуете тему, анализируете данные, пишете отчёт |
| **`fusion-agents`** | запускаете автономный агентный цикл на много шагов |

### Или пропустите всё это — одна команда

Скрипты из комплекта поднимают прокси, если он не запущен, сами пишут конфигурацию клиента и открывают ваш инструмент:

```bash
./bin/fusion-opencode fusion-coder     # OpenCode TUI, подключено и готово
./bin/fusion-claude fusion-agents      # Claude Code, подключено и готово
```

## Подключение к вашему инструменту

### OpenCode

Самый быстрый путь — скрипт запуска выше. Чтобы подключить вручную, добавьте блок провайдера в `opencode.json` (локально в проекте или глобально `~/.config/opencode/opencode.json`):

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "fusion": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Fusion Proxy",
      "options": { "baseURL": "http://127.0.0.1:8080/v1" },
      "models": {
        "fusion-coder": { "name": "Fusion Coder" },
        "fusion-researcher": { "name": "Fusion Researcher" },
        "fusion-agents": { "name": "Fusion Agents" }
      }
    }
  }
}
```

Идентификаторы моделей должны совпадать с виртуальными именами в вашем `fusion.yaml`. Если вы задали `FUSION_PROXY_TOKEN` (аутентификация клиентов), добавьте токен для провайдера с id `fusion` через `opencode auth login` (или ваш `auth.json`), чтобы OpenCode отправлял его как Bearer-токен.

Когда прокси запущен, модель выбирается через `-m fusion/<name>`:

```bash
opencode -m fusion/fusion-coder                         # интерактивный TUI, код / планирование
opencode -m fusion/fusion-agents                        # интерактивный TUI, автономная маршрутизация
opencode run -m fusion/fusion-researcher "Summarize X"  # без интерфейса, один запрос
```

Скрипт `fusion-opencode` делает все три шага — поднимает прокси при необходимости, записывает провайдера `fusion` в вашу глобальную конфигурацию OpenCode (идемпотентно: он трогает только ключ `fusion`) и запускает OpenCode:

```bash
./bin/fusion-opencode fusion-researcher        # TUI для исследований / отчётов
./bin/fusion-opencode fusion-agents run "Fix the failing test in src/foo.ts"   # разовый запуск без интерфейса
```

Добавьте его в `PATH`, чтобы вызывать из любой папки:

```bash
npm link            # или: ln -s "$PWD/bin/fusion-opencode" ~/.local/bin/fusion-opencode
fusion-opencode fusion-coder
```

Скрипт учитывает `FUSION_PROXY_URL` (по умолчанию `http://127.0.0.1:8080`) и, если ваш прокси требует аутентификацию клиентов, `FUSION_PROXY_TOKEN` (используется как apiKey, который отправляет OpenCode).

### Claude Code

Claude Code использует **Anthropic Messages API**. llm-fusion открывает `POST /v1/messages` на том же базовом адресе, так что `bin/fusion-claude` поднимает прокси (при необходимости), экспортирует переменные окружения Anthropic и запускает Claude Code с выбранной моделью:

```bash
./bin/fusion-claude fusion-agents            # автономные агентные циклы (по умолчанию)
./bin/fusion-claude fusion-coder             # программирование / планирование
./bin/fusion-claude fusion-researcher        # исследования / отчёты
./bin/fusion-claude fusion-agents run "Fix the failing test in src/foo.ts"
```

Он учитывает `FUSION_PROXY_URL` и `FUSION_PROXY_TOKEN` точно так же, как `fusion-opencode`. Одно отличие от OpenAI-совместимых клиентов: Claude Code использует **корень** прокси (`http://127.0.0.1:8080`), а не `/v1`, потому что обращается к `/v1/messages`.

Полное руководство по настройке, справочник переменных окружения и разбор проблем: [`docs/claude-code.md`](./docs/claude-code.md).

### Любой другой OpenAI-совместимый клиент

Прокси — это готовая точка входа OpenAI Chat Completions, так что работает всё, что говорит на языке OpenAI: Continue, Cline, Aider с базовым адресом OpenAI или ваш собственный цикл:

- **Базовый адрес:** `http://127.0.0.1:8080/v1`
- **Ключ API:** любая непустая строка, пока прокси без аутентификации (например `local-no-auth`); реальное значение `FUSION_PROXY_TOKEN`, если вы включили аутентификацию клиентов.
- **Модель:** `fusion-coder`, `fusion-researcher` или `fusion-agents` — см. [Какую модель выбрать?](#какую-модель-выбрать)

## Обновление

`llm-fusion` живёт в git и не публикуется в npm, так что «обновиться» — это подтянуть последние изменения и освежить зависимости:

```bash
git pull                          # перемотка к новейшему коммиту / тегу
npm install                       # подхватить изменения зависимостей
```

Затем перезапустите прокси. Ваши `fusion.yaml` и `.env` — ваши: `git pull` никогда не перезаписывает локальные изменения в них (в комплектном `fusion.yaml` лежат пресеты моделей, которые вы могли редактировать, — так что если редактировали, либо закоммитьте/спрячьте изменения, либо держите их в копии, на которую указывает `FUSION_CONFIG`). Чтобы зафиксироваться на конкретном релизе вместо ветки `main`:

```bash
git fetch --tags
git checkout v0.1.27              # или любой тег со страницы Releases
npm install
```

`fusion.example.yaml` — полностью аннотированный справочник, и он обновляется между релизами: когда выходит новая версия, сравните его со своей рабочей конфигурацией, чтобы увидеть новые опции. Журнал изменений: [`CHANGELOG.md`](./CHANGELOG.md), релизы: <https://github.com/Lexus2016/LLM-Fusion/releases>.

# Технический справочник

Дальше — детали: как работает конвейер, сколько он стоит, каждый ключ конфигурации, каждый адрес. Для старта ничего из этого не нужно — быстрый старт выше и есть вся настройка.

## Какую модель выбрать?

В `fusion.yaml` есть три **специализированных под задачи** пресета, каждый собран по результатам эмпирического отбора моделей (8 моделей × 3 тестовые задачи — код, исследования, вызовы инструментов):

| Вызывайте эту модель | Для чего | Стратегия | Как собрана |
|---|---|---|---|
| **`fusion-coder`** | программирование, планирование, аудит кода | `fusion` | панель `kimi-k2.7-code` + `deepseek-v4-flash` + `gemini-3-flash-preview` + `qwen3-coder-next` → судья `glm-5.2` → синтезатор `glm-5.2` |
| **`fusion-researcher`** | исследования, анализ, отчёты | `fusion` | панель `kimi-k2.7-code` + `glm-5.2` + `gpt-oss:120b` → судья `glm-5.2` → синтезатор `kimi-k2.7-code` |
| **`fusion-agents`** | автономные агентные циклы | `smart` | маршрутизатор `glm-5.2`; простые шаги → `glm-5.2`, сложные шаги и восстановление после ошибок → панель `fusion-coder` |

Два правила выросли прямо из данных:

- **Для кода — fusion, а не одна модель.** Архитектура и планирование действительно выигрывают от нескольких точек зрения. (Чистый *аудит* кода — просто перечисление проблем — единственная «кодовая» задача, где одна модель побеждает, и она не репрезентативна для программирования.)
- **Панели намеренно смешивают родословные моделей.** `glm`/`kimi`/`deepseek`/`minimax`/`qwen` — всё китайские лаборатории с общими слепыми зонами; каждая панель добавляет западный декоррелятор — `gemini-3-flash-preview` (Google) в панели кодера и `gpt-oss:120b` (родословная OpenAI) в панели исследователя, — чтобы ошибки панели были менее коррелированными; в этом весь смысл панели. (Gemini был убран в v0.1.27 из-за ошибки `thought_signature` на чужой истории вызовов инструментов, но spot-check 2026-07-08 с синтетической многоходовой историей инструментов не воспроизвёл ошибку, поэтому он вернулся в v0.1.28; в реальных агентных циклах следите за строкой `tool-turn terminal state` в логе.)

Изначальные универсальные пресеты (`fusion-1`, `smart-1`, `fast-glm` / `fast-kimi` / `fast-deepseek`) по-прежнему в комплекте для разовых нужд.

## Архитектура

Запрос на `POST /v1/chat/completions` несёт **виртуальное имя модели**. Прокси сопоставляет это имя с одной из четырёх стратегий и исполняет её:

| Стратегия | Что делает |
|----------|--------------|
| `single` | Прямая передача 1:1 в одну upstream-модель (стрим и без стрима). Примитив, из которого построено всё остальное. |
| `failover` | Упорядоченная цепочка; при сбое до первого токена переключается на следующего участника. Устойчивость без разветвления. |
| `fusion` | Конвейер обдумывания: **панель → судья → синтезатор** (ниже). |
| `smart` | LLM-**маршрутизатор** классифицирует каждый запрос и направляет его в `single` (дёшево) или `fusion` (глубоко). |

### Fusion: панель → судья → синтезатор

1. **Панель** — запрос идёт к N моделям *параллельно* (по умолчанию 3). В режиме инструментов `deliberate` участники панели получают *описания* инструментов обычным текстом, но не реальную схему `tools`, поэтому дальше всегда идёт только один канонический вызов инструмента. По желанию один участник работает **состязательным рецензентом** (мандат «красной команды»: найди изъян, отстаивай противоположную позицию) — см. «Известные ограничения». Длинные контексты сжимаются перед разветвлением панели, чтобы не переполнить контекстные окна участников.
2. **Судья** — один вызов со структурированным JSON сравнивает ответы панели (консенсус, разногласия, уникальные наблюдения, слепые зоны, частичное покрытие, hallucination_flags) и выдаёт **калиброванную `confidence`** (`high`/`medium`/`low`) плюс `fragile_claims`; синтезатор смягчает хрупкие и малодостоверные утверждения вместо того, чтобы выдавать общие предубеждения за уверенность. Если судья возвращает некорректный JSON, этап мягко деградирует к сырым ответам панели, а не падает.
3. **Синтезатор** — финальная модель пишет ответ, который получает клиент, стримит, когда просят, и **единственная** получает реальную схему `tools` (так что до агента доходит ровно один `tool_calls`). Опционально после него идёт оценка фактической согласованности **BinEval**, возвращаемая в заголовках.

### Smart: маршрутизатор

`smart` делает **один** нестримовый, `temperature: 0`, чисто JSON-вызов к быстрой модели-`router`, читает `{ "route": "simple" | "fusion" }` и направляет запрос в соответствующую подстратегию. Любая ошибка маршрутизатора, таймаут или неразборчивый ответ откатываются к настроенному маршруту `default` (по умолчанию `simple`, потому что контроль расходов — это и есть суть). Подстратегии могут быть встроенными блоками или строковыми ссылками на другие настроенные модели, чтобы не повторяться. Решения маршрутизатора кешируются для идентичных тел запросов, а одновременные идентичные запросы объединяются, так что всплеск одинаковых ходов повторно использует один раунд маршрутизатора. Если последний результат инструмента выглядит как сбой (error / exception / ненулевой код выхода), запрос эскалируется прямо в `fusion`, минуя маршрутизатор: восстановление — именно то место, где обдумывание окупается.

### Необязательные функции fusion

Все они **по умолчанию ВЫКЛЮЧЕНЫ**; включаются для каждой fusion-модели отдельно в `fusion.yaml`. Полный справочник: [`fusion.example.yaml`](./fusion.example.yaml).

- **Калиброванная уверенность судьи** — судья выдаёт `confidence` (`high`/`medium`/`low`) и `fragile_claims` (спорные / подтверждённые лишь одним источником / слабые утверждения). Синтезатор смягчает хрупкие и малодостоверные утверждения вместо того, чтобы выдавать общие предубеждения из обучения за уверенность. Судье **предписано инструкцией**, что любые `hallucination_flags` или `fragile_claims` должны снижать `confidence` до `medium`/`low` (это правило на уровне промпта, которому модель-судья должна следовать, а не жёсткая проверка в коде — схема принимает JSON судьи как есть).
- **Состязательный участник панели** — `adversarial: <panel-member>` превращает одно место в рецензента «красной команды»: отстаивать противоположную позицию, искать изъяны, скрытые допущения, крайние случаи и состояния гонки вместо согласия с консенсусом. Ролевая декорреляция поверх декорреляции по родословной. Состязательный участник **никогда не отменяется досрочно**, панель его ждёт, так что его ответ никогда не теряется; ему сказано отвечать «I cannot find a real problem», когда консенсус прочен. Проверяется, что это существующий участник панели.
- **Веб-заземление (Tavily)** — `web_search: { enabled: true, max_results, timeout_s, max_context_chars, max_prompt_chars }`. Прокси выполняет **один** поиск Tavily перед разветвлением панели и вставляет очищенные результаты как текстовый контекст каждому участнику (никто не получает реальных инструментов, так что инвариант «один `tool_calls`» сохраняется). Условия включения: `TAVILY_API_KEY` в окружении *и* `web_search.enabled` на модели (нет ключа и нет согласия → нет поиска, задержки и расходов), **плюс два пропуска во время исполнения**: если в последнем сообщении пользователя нет пригодного текста запроса или промпт уже превышает `max_prompt_chars` (по умолчанию 80 000) — ограничение по размеру не даёт веб-контексту переполнить участника панели с меньшим окном (например, gpt-oss:120b со 128k) в длинных продолжениях агентного цикла. Контекст вставляется как ход `user` (не `system`) и начинается с текущей даты, чтобы модели с устаревшей границей знаний (kimi-k2.7-code) пользовались свежими фактами вместо отказа. Неудавшийся поиск тихо деградирует к незаземлённой панели. Уважает `fusion_planning_turn_only`. Ключ — в `.env`:
  ```bash
  echo 'TAVILY_API_KEY=tvly-...' >> .env
  ```
- **Пост-синтезная проверка качества BinEval** — `bineval: { enabled: true, model: <eval>, threshold: 0.7, dimensions: [...] }`. После успешного *нестримового* синтеза один дополнительный вызов оценщика выставляет ответу балл за фактическую согласованность (или по вашим собственным бинарным вопросам) и возвращает результаты в заголовках ответа: `X-Fusion-Bineval-Score` (0–1), `X-Fusion-Bineval-Dimensions` (JSON по измерениям) и `X-Fusion-Bineval-Low-Score: true`, когда общий балл ниже `threshold`. Когда bineval настроен, но оценка не состоялась, прокси ставит `X-Fusion-Bineval-Skipped: <причина>`, чтобы клиент мог отличить «балл высокий» от «оценки не было» — причины: `streaming`, `synth_error` (синтез ≥400), `eval_failed` (оценщик ошибся / вышел за время / разомкнута цепь), `empty_output` (ответ только с инструментами), `non_json_body`, `synth_only` (путь planning-turn-only внутри цикла / обход). **BinEval только отчитывается** — он не запускает перемаршрутизацию или повторное обдумывание.

### Надёжность агентных циклов (v0.1.16+)

Долгие агентные прогоны раньше погибали на четырёх зависящих от нагрузки режимах сбоя; теперь все четыре закрыты структурно:

- **Защита полноты синтеза с запасным судьёй.** «Мыслящая» модель-синтезатор может объявить себя завершённой, ещё находясь посреди плана (пустой ответ или один встроенный блок `<think>` без артефакта). Защита обнаруживает это и на стримовом, и на нестримовом путях, повторяет тот же синтез один раз со строгим требованием завершить и — если повтор так же непригоден (или сам обрезан лимитом токенов посреди вызова инструмента) — делает ОДНУ попытку на другой модели (судье, а когда судья и есть синтезатор — участнике панели), так что ни одна отдельная модель не может остановить агентный цикл. Во время тихого восстановления стрим несёт SSE-комментарии `: keepalive` (каждые 5 с, настраивается через `FUSION_SYNTH_RECOVERY_PING_MS`), чтобы клиенты и посредники не отваливались по таймауту.
- **Честный `stop_reason` при обрезании лимитом токенов.** Вызов инструмента, обрезанный `max_tokens`, сообщается Anthropic-клиентам как `stop_reason: "max_tokens"` — а не как якобы пригодный к исполнению `tool_use` со сломанным JSON на входе. Когда лимит приходится точно *после* завершения JSON вызова, вызов остаётся пригодным (`tool_use`), так что завершённый многоминутный Write никогда не выбрасывается.
- **Бюджеты конкурентности на модель.** `upstream.per_model_concurrency` даёт каждой реальной upstream-модели собственный шлюз перед глобальным ограничителем: всплеск фоновых вызовов маленькой модели (Claude Code выдаёт 80–130/мин) выстраивается в собственную очередь вместо блокировки интерактивных ходов в начале общей.
- **Разделённые классы трафика.** Скрипт `fusion-claude` по умолчанию ставит фоновую модель Claude Code (`ANTHROPIC_SMALL_FAST_MODEL`) в `fast-deepseek` — модель, от которой не зависит ни одна панель, судья, синтезатор, маршрутизатор или простой маршрут, — так что фоновые всплески не могут ограничить скорость модели, которая пишет ваши файлы.
- **Защита инструментальных ходов одиночного маршрута (v0.1.26).** Прямая передача smart `simple` — маршрут, которым идёт ~87 % агентных шагов, — получила собственную защиту полноты. Три подтверждённых на практике режима сбоя раньше завершали ход агента без единого выполненного действия (то самое состояние «сделал один шаг и стоит, пока не наберёте *continue*»): модель *пересказывает* следующее действие прозой вместо выдачи вызова инструмента; запись большого файла обрезается лимитом вывода посреди аргументов (`finish_reason:"length"`, непригодный JSON); upstream рвёт длинные потоки генерации (~5 мин на Ollama Cloud). Защита обнаруживает все три на запросах с инструментами и выполняет один повтор восстановления **с живым стримом** (с подсказкой выдать вызов инструмента и писать большие полезные нагрузки кусками), а при неудаче отдаёт изначальный ответ. Каждый стрим с инструментами также пишет в лог одну строку `tool-turn terminal state`, так что реальное зависание сессии диагностируется по одному только логу.
- **`request_overrides` на модель (v0.1.26).** Одиночные модели (и встроенный слот smart `simple`) принимают дополнительные поля тела запроса, подмешиваемые в каждый upstream-вызов — например, `request_overrides: { reasoning_effort: "none" }` останавливает «мыслящую» модель от многоминутных размышлений на механических агентных шагах (измерено A/B на glm-5.2: рассуждения 1692→0 символов, 6 с→2 с, вызовы инструментов не задеты; `think:false` и `"low"` Ollama Cloud игнорирует). Ключевые поля (`model`, `messages`, `stream`, `tools`, `tool_choice`) защищены. Комплектный пресет `fusion-agents` использует именно это.

## Коннекторы, мультиаккаунтный failover и панель

По умолчанию проксирование идёт через один upstream (один аккаунт Ollama
Cloud). Вместо этого можно сгруппировать upstream'ы в **провайдеры**, каждый со
своими **аккаунтами**, и прокси **автоматически переключается между аккаунтами
одного провайдера**, когда один деградирует, исчерпал лимит или у него
закончилась оплата. Failover остаётся *внутри* провайдера (те же модели), а
каждая виртуальная модель привязана к одному провайдеру — поэтому fusion никогда
не «перепрыгнет» тихо на провайдера с другим каталогом моделей.

```yaml
providers:
  ollama-cloud:                 # группа-провайдер = один провайдер + его аккаунты
    type: ollama
    base_url: https://ollama.com
    accounts:                   # failover между ними (те же модели, разные ключи)
      - { id: ollama-1, api_key_env: OLLAMA_API_KEY }
      - { id: ollama-2, api_key_env: OLLAMA_API_KEY_2 }
  openrouter:                   # другой провайдер = своя отдельная группа
    type: openai-compat
    base_url: https://openrouter.ai/api/v1
    accounts:
      - id: openrouter-1
        api_key_env: OPENROUTER_API_KEY
        model_map: { "qwen3-coder:480b": "qwen/qwen3-coder" }

models:
  # привяжите модель к группе-провайдеру (необязательно, когда группа одна)
  fusion-coder: { strategy: fusion, provider: ollama-cloud, panel: [glm-5.2, kimi-k2.7-code], judge: glm-5.2, synth: kimi-k2.7-code }
```

**Как решается failover.** Обслуживает первый здоровый аккаунт. `429`
(лимит), `5xx`, сетевая ошибка или таймаут **охлаждают** этот аккаунт на
`connector_cooldown_s`, и эстафету принимает следующий аккаунт того же
провайдера; охлаждённый аккаунт автоматически проверяется проб-запросом после
окна ожидания. `401` (плохой ключ), `402` (закончились средства) или совпадение
маркера квоты в теле помечают его как **down** («оплата закончилась») — он
пропускается до `connector_down_recheck_s` или ручного сброса. Когда
задросселировано несколько аккаунтов, прокси отдаёт наверх наиболее восстановимую
ошибку (временный `429` важнее `401` мёртвого backup), поэтому успешный запрос
никогда не превращается в жёсткий сбой. Это прозрачно: каждая модель/стратегия
(single, failover, fusion, smart) едет на пуле своего провайдера без изменений
конфига.

**Любой OpenAI-совместимый провайдер — только конфигом.** Обобщённый тип
`openai-compat` покрывает OpenRouter, DeepInfra, Together, Novita, Nebius, Groq,
Cerebras, DeepSeek, Mistral, Baseten, … — задайте `base_url`, `api_key_env` и
`model_map` (их ID моделей отличаются от Ollama). Ранжированное сравнение и то,
какие провайдеры чисто сигналят об исчерпанном аккаунте (`402`), — в
[`docs/providers-research.md`](docs/providers-research.md).

**Панель — мониторинг + редактор без YAML.** Откройте
**http://127.0.0.1:8080/panel**. У неё три вкладки:

- **Monitor** — здоровье вживую: какой аккаунт активен, какие охлаждаются или
  выключены и *почему* (причина + последняя ошибка + обратный отсчёт), счётчики
  запросов/ошибок на аккаунт и ручное управление (выключить, включить, сбросить
  или закрепить аккаунт активным). Без мерцания — вид обновляется на месте, а не
  перерисовывается заново.
- **Providers** — добавляйте, редактируйте или удаляйте группы-провайдеры и их
  аккаунты через формы с подсказками простым языком. Никакого YAML руками.
- **Models** — так же создавайте и редактируйте модели `fusion` / `smart` /
  `failover` / `single`: выберите стратегию, и форма покажет только нужные ей
  поля (panel, judge, synth, router, …), каждое с пояснением рядом.

Форма **Models** не заставляет вводить ID моделей по памяти: она подтягивает
**живой каталог** выбранного провайдера (`GET /v1/models`, с откатом на
`/api/tags` Ollama) и предлагает его списком выбора — поэтому участники
panel / judge / synth / router **выбираются, а не вводятся с ошибкой**. Каждая
правка проверяется той же схемой, с которой стартует сервер, записывается
редактором YAML с сохранением комментариев, резервируется (с временной меткой) и
сохраняется атомарно; модели подхватываются «на горячую», а изменения
провайдеров перестраивают живой пул — **без перезапуска**. Деструктивные
действия (выключение / удаление / …) сначала спрашивают подтверждение. Когда
задан клиентский auth-токен, данные и действия панели требуют его (HTML-оболочка
не содержит секретов). Обратная совместимость: без `providers:` работает как
раньше — старый одиночный `upstream.base_url` + `api_key_env`.

## Честная заметка о стоимости (прочитайте это)

Полный fusion работает на **каждом** шаге. Агентный цикл (прочитать → подумать → отредактировать → прогнать тесты → перечитать…) умножает количество upstream-**вызовов моделей**:

```
upstream_calls_per_step = N_panel + 1 (судья) + 1 (синтезатор) = 3 + 1 + 1 = 5
```

Типичная задача с кодом на **15–25 шагов** поэтому выдаёт примерно **75–125 upstream-вызовов моделей**, если fusion включён всегда.

- **Исполнение инструментов НЕ умножается** — инструменты запускаются раз на шаг (канонический `tool_calls` выдаёт только синтезатор). Действия вашего агента не затронуты.
- **Вызовы API моделей умножаются** — на множитель панель + судья + синтезатор. Это и есть цена, и для v1 она принята осознанно.

**Смягчение (в комплекте, не «на потом»): стратегия `smart`.** Направьте агента на `fusion-agents` — и рутинные шаги (`read_file`, `grep`, …) пойдут дешёвым путём `single` (1 вызов), а действительно сложные всё так же получат полный fusion. Другие встроенные рычаги:

- маленькая панель по умолчанию (3),
- глобальный лимит `max_concurrency` плюс **бюджеты конкурентности на модель** (`per_model_concurrency`) — всплеск на одной модели выстраивается в собственную очередь вместо блокировки интерактивных ходов,
- жёсткие таймауты панели/судьи (90 с / 120 с в комплектных пресетах, оба ниже upstream-потолка ~182 с),
- переключатель `fusion_planning_turn_only` (полная панель на каждом плановом ходе — любом запросе, где последнее сообщение является свежей инструкцией пользователя, — и деградация до одного лишь синтеза — 5 вызовов → 1 — только на продолжениях с результатами инструментов внутри цикла; новая задача глубоко в длинной сессии всё равно получит панель).

Именно так llm-fusion держит долгие агентные циклы доступными по деньгам, не жертвуя обдумыванием там, где оно важно.

## llm-fusion против OpenRouter Fusion

Оба прогоняют запрос через панель моделей плюс шаг судьи/синтезатора, чтобы свести несколько взглядов в один ответ. Разница — в том, **где, когда и как** работает этот конвейер.

| | **llm-fusion** | **OpenRouter Fusion** |
|---|---|---|
| **Размещение и контроль** | Работает на вашей машине / в Docker. Один пользователь, можно заглянуть внутрь, один файл конфигурации. | Полностью управляемый SaaS на OpenRouter. |
| **Счёт** | Платите Ollama Cloud напрямую. | Платите OpenRouter; использование идёт пакетом. |
| **Охват провайдеров** | Только Ollama Cloud (один счёт, один upstream). | Любой провайдер на OpenRouter (более широкий каталог моделей). |
| **Автоматическая маршрутизация** | Встроенная стратегия `smart`: быстрый LLM-маршрутизатор выбирает `single` (дёшево) или `fusion` (глубоко) **для каждого запроса**. | Fusion всегда гонит полную панель; маршрутизация ручная или отдельным маршрутизатором. |
| **Безопасность агентных циклов** | Выдаёт **ровно один `tool_calls`** на шаг; панель никогда не касается реальных инструментов в режиме `deliberate`. | Плагин возвращает анализ; финальное использование инструментов решает вызывающая модель. |
| **Рычаг стоимости для долгих прогонов** | `fusion_planning_turn_only`: полная панель на плановом ходе, дальше только синтез (5 вызовов → 1) для механических шагов внутри цикла. | Нет автоматического понижения отдельных запросов внутри цикла. |
| **Веб-заземление** | Опциональное, по явному согласию: один поиск Tavily перед панелью, результаты вставляются текстовым контекстом. Условия: `TAVILY_API_KEY` (окружение) **и** `web_search.enabled` (конфигурация) — полностью ВЫКЛЮЧЕНО, пока не заданы оба. | Встроенные `web_search` + `web_fetch` для панели и судьи. |
| **Устойчивость** | Цепочки `failover` + мягкая деградация + лимит `max_concurrency`. | Управляемая надёжность + защита от рекурсии. |
| **Стиль конфигурации** | Один YAML с горячей перезагрузкой и эмпирически подобранными пресетами. | JSON плагина / серверный инструмент / slug модели; пресеты + веб-интерфейс. |

**Берите llm-fusion, когда:**

- Вы гоняете автономного агента **сотни шагов** и вам нужны детерминированные побочные эффекты ровно-один-раз.
- Вы хотите, чтобы рутинные шаги стоили **1 upstream-вызов**, а fusion оставался для сложных шагов и восстановления после ошибок.
- Вам ближе **локальный, прозрачный процесс** с одним файлом конфигурации и без дополнительной привязки к поставщику.
- Ваши модели и бюджет живут на **Ollama Cloud**.

**Берите OpenRouter Fusion, когда:**

- Нужны **исследования с веб-заземлением** на передовых моделях многих провайдеров.
- Хотите **управляемый сервис** с веб-площадкой и глобальной инфраструктурой.
- Вы делаете разовое глубокое исследование, а не долгие агентные циклы.

**Суть:** OpenRouter Fusion оптимизирует потолок качества разового ответа на исследовательских задачах. **llm-fusion оптимизирует работу внутри агентного цикла** — контроль расходов, предотвращение двойных вызовов инструментов и тяжёлое обдумывание только там, где оно действительно окупается.

## Конфигурация

По умолчанию прокси загружает `./fusion.yaml` (путь меняется через `FUSION_CONFIG`). Вот сокращённая копия комплектной конфигурации — три специализированных пресета плюс одна single-модель:

```yaml
upstream:
  base_url: https://ollama.com
  api_key_env: OLLAMA_API_KEY        # ИМЯ переменной окружения с ключом — никогда не сам ключ
server:
  bind: 127.0.0.1
  port: 8080
models:
  # single — прямая передача 1:1; вызывайте, когда нужна одна быстрая модель без панели
  fast-glm:
    strategy: single
    target: glm-5.2

  # fusion — панель -> судья -> синтезатор; программирование, планирование, аудит кода
  fusion-coder:
    strategy: fusion
    panel: [kimi-k2.7-code, deepseek-v4-flash, gemini-3-flash-preview, qwen3-coder-next]
    judge: glm-5.2
    synth: glm-5.2
    tool_mode: deliberate

  # fusion — панель под исследования и отчёты, опциональное веб-заземление
  fusion-researcher:
    strategy: fusion
    panel: [kimi-k2.7-code, glm-5.2, gpt-oss:120b]
    judge: glm-5.2
    synth: kimi-k2.7-code
    tool_mode: deliberate
    web_search:
      enabled: true                  # также нужен TAVILY_API_KEY в .env, иначе полностью ВЫКЛЮЧЕНО
      max_results: 3

  # smart — маршрутизатор выбирает дешёвый single или полный fusion на каждый запрос; агентные циклы
  fusion-agents:
    strategy: smart
    router: glm-5.2
    default: simple
    escalate_on_tool_error: true     # сбойный результат инструмента идёт прямо в fusion
    simple:
      target: glm-5.2                # механические шаги (read / grep / run)
      request_overrides:
        reasoning_effort: "none"     # без многоминутных размышлений на механических шагах
    fusion: fusion-coder             # сложные шаги -> панель кодера (строковая ссылка)
```

Вызывается любое из этого через поле `model` в запросе — `fast-glm` ответит одной моделью без панели. Полный комплектный [`fusion.yaml`](./fusion.yaml) дополнительно содержит `fast-kimi` / `fast-deepseek` / `fusion-1` / `smart-1` для разовых нужд, подобранные таймауты, бюджеты конкурентности на модель и блоки BinEval. Всё, что не задано, откатывается к задокументированным значениям по умолчанию. **Полностью аннотированный справочник** — каждый ключ, каждое значение по умолчанию, `failover`, `tool_mode`, `fusion_planning_turn_only`, `overrides`, встроенные подблоки smart — см. [`fusion.example.yaml`](./fusion.example.yaml).

Шпаргалка стратегий для карты `models:`:

- **`single`** → `{ strategy: single, target: <model> }`
- **`failover`** → `{ strategy: failover, chain: [<m1>, <m2>, …] }`
- **`fusion`** → `{ strategy: fusion, panel: [...], judge: <model>, synth: <model> }`
- **`smart`** → `{ strategy: smart, router: <model>, default: simple|fusion, simple: <single-блок-или-ссылка>, fusion: <fusion-блок-или-ссылка> }`

Конфигурация **перезагружается горячо**: отредактируйте `fusion.yaml` — изменения маршрутизации/моделей применятся вживую, а правки `providers:` перестроят пулы коннекторов на лету (некорректная правка отклоняется, предыдущая конфигурация сохраняется). Только параметры уровня процесса из блока `upstream:` — `max_concurrency`, `request_timeout_s`, паузы коннекторов — читаются один раз при старте и требуют перезапуска.

## Переменные окружения

Прокси читает обычные переменные окружения. Он также **автоматически подгружает локальный `.env`** на старте (нативно в Node 24; отсутствие файла — не проблема).

| Переменная | Обязательная | По умолчанию | Назначение |
|----------|----------|---------|---------|
| `OLLAMA_API_KEY` | Да, для живой работы | — | Bearer-ключ Ollama Cloud. Имя — то, на что указывает `upstream.api_key_env` (по умолчанию `OLLAMA_API_KEY`). Держится только на стороне сервера; никогда не отправляется клиентам и не попадает в лог. |
| `FUSION_PROXY_TOKEN` | Нет | не задано | Когда в конфигурации задано `server.auth_token_env: FUSION_PROXY_TOKEN`, клиенты обязаны отправлять `Authorization: Bearer <это значение>`. Если переменная указана в конфигурации, но НЕ задана (например, опечатка в имени), аутентификация не отключается молча: каждый запрос получает 500, а при старте пишется ошибка. |
| `FUSION_CONFIG` | Нет | `./fusion.yaml` | Путь к файлу конфигурации. |
| `LOG_PRETTY` | Нет | не задано | `LOG_PRETTY=1` включает удобные для чтения логи (иначе — структурированный JSON). |
| `LOG_LEVEL` | Нет | `info` | Уровень логов pino (`debug`, `info`, `warn`, …). |
| `FUSION_BIND` | Нет | `server.bind` | Переопределяет адрес прослушивания без правки конфигурации (используется Docker-образом). |
| `FUSION_ALLOW_OPEN` | Нет | не задано | Прокси отказывается запускаться на не-loopback адресе без аутентификации клиентов (например, Docker без `FUSION_PROXY_TOKEN`). Задайте `FUSION_ALLOW_OPEN=1`, чтобы переопределить — только за вашим собственным шлюзом аутентификации. |
| `FUSION_SYNTH_RECOVERY_PING_MS` | Нет | `5000` | Интервал SSE-комментариев `: keepalive`, идущих в стрим, пока выполняется повтор восстановления синтеза. Положительное число; всё прочее откатывается к значению по умолчанию. |

`.env.example` здесь закоммитить нельзя (ограничение песочницы). Создайте собственный `.env` в корне проекта с таким буквальным содержимым:

```dotenv
# Обязательно для любой живой (не тестовой) работы:
OLLAMA_API_KEY=ollama-your-key-here

# Опционально — раскомментируйте, чтобы требовать клиентский токен (также задайте
# server.auth_token_env: FUSION_PROXY_TOKEN в fusion.yaml):
# FUSION_PROXY_TOKEN=choose-a-long-shared-secret

# Опциональные переопределения:
# FUSION_CONFIG=./fusion.yaml
# LOG_PRETTY=1
```

## Запуск

```bash
npm run dev      # tsx watch — перезапуск при изменении кода
npm run start    # tsx — разовый запуск
```

Или с ключом прямо в команде (без `.env`):

```bash
OLLAMA_API_KEY=ollama-... npm run start
```

На старте печатается баннер: адрес прослушивания, загруженные виртуальные модели с их стратегиями, включена ли аутентификация клиентов и присутствует ли upstream-ключ.

## Endpoints

| Метод и путь | Назначение |
|---------------|---------|
| `POST /v1/chat/completions` | Главная точка входа. Совместима с OpenAI; поддерживает `stream`, `tools`, `tool_choice` и блоки изображений. Маршрутизируется по виртуальному имени `model`. |
| `POST /v1/messages` | Точка входа Anthropic Messages API для Claude Code. Переводит блоки содержимого Anthropic во внутренний OpenAI-конвейер и обратно; поддерживает стриминг и инструменты. |
| `GET /v1/models` | Перечень настроенных виртуальных моделей (формат списка OpenAI). Добавляет `context_window` / `supports_vision` там, где возможности модели известны. |
| `GET /health` | Живучесть. `200`, если процесс жив. Без проверки upstream. |
| `GET /ready` | Готовность. `200` только если upstream достижим и репрезентативная модель находится; иначе `503`. |
| `GET /panel` | Локальная панель (HTML): 3 вкладки — Monitor (здоровье), Providers и Models (редактор без YAML). Показывает, какой аккаунт активен, какие охлаждаются/выключены и почему, с ручным управлением. |
| `GET /admin/providers` | Групповой JSON-снимок здоровья провайдеров/аккаунтов (под auth-токеном, если задан). Питает панель. |
| `POST /admin/connectors/:id/{disable,enable,reset,pin,unpin}` | Ручное управление аккаунтами (под auth-токеном). |
| `GET /admin/config` | Текущий редактируемый конфиг — группы-провайдеры, модели, defaults и то, какие env-переменные с ключами присутствуют (только **имена** env-переменных + наличие, никогда не значения секретов). Питает вкладки Providers/Models. |
| `PUT · DELETE /admin/config/models/:name` | Создать/изменить или удалить виртуальную модель. Проверено, с сохранением комментариев, с резервированием, атомарно; подхватывается «на горячую». |
| `PUT · DELETE /admin/config/providers/:id` | Создать/изменить или удалить группу-провайдера. Те же гарантии записи; перестраивает живой пул. |
| `GET /admin/config/providers/:id/models` | Живой каталог моделей провайдера (для выбора без ошибок). Кратко кэшируется; при сбое деградирует до пустого списка, поэтому форма не блокируется. |

## Фаза 0 — живая проверка

Корректность адаптера upstream держится на допущениях об Ollama Cloud (Bearer + SSE, вызовы инструментов для каждой модели, обнаружение через `/api/show`, формат изображений). Проверьте их на живом API своим ключом:

```bash
OLLAMA_API_KEY=ollama-... npm run smoke
```

Это запускает `test/live.smoke.test.ts` против `https://ollama.com`. Каждая проверка — отдельный тест с чётким результатом в логе:

- **A-1 — Bearer + ответ** *(ПРОВЕРЕНО 2026-06-26)*: нестримовый чат к `glm-5.2` возвращает ответ (подтверждает ключ и openai-совместимый путь).
- **SSE — стриминг** *(ПРОВЕРЕНО 2026-06-26)*: тот же запрос со `stream: true` возвращает SSE-куски `data:`.
- **A-2 — вызовы инструментов по моделям** *(ПРОВЕРЕНО 2026-06-26 — `tool_calls` на всех: `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`)*: запрос с `tools` к каждой из `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`; в лог идёт **PASS/FAIL** для каждой (сьют *не* падает, если модель без инструментов — но синтезатором должна быть модель с PASS, потому что весь конвейер обдумывания зависит от её `tool_calls`).
- **A-3 — `/api/show`** *(ПРОВЕРЕНО 2026-06-26 — `capabilities[]` + `*.context_length` вернулись для всех трёх)*: подтверждает, что для этих моделей приходят `capabilities[]` и `*.context_length`, и логирует найденное.
- **A-4/A-5 — изображения** *(ЕЩЁ ОТКРЫТО — перепроверить на `kimi-k2.7-code`; предыдущую тестовую модель `qwen3-vl:235b` сняли 2026-06-16)*: отправляет PNG 1×1 как `image_url` модели со зрением (`kimi-k2.7-code` по умолчанию; меняется через `FUSION_VISION_MODEL`) и логирует, принят ли формат изображений OpenAI.

Сьют печатает итог, по которому вы фиксируете две настройки в `fusion.yaml`: `upstream.api_mode` (`openai`, если проверка изображений приняла формат OpenAI, иначе `native`) и какую модель безопасно ставить fusion-`synth` (ту, чья строка A-2 — PASS).

Без `OLLAMA_API_KEY` сьют **пропускается**, так что он никогда не бегает в CI или типовом прогоне тестов.

## Тестирование

```bash
npm test          # vitest run — быстро, офлайн, без ключа и сети (живой smoke пропускается)
npm run typecheck # tsc --noEmit
```

Модульный/интеграционный сьют использует мок-upstream (перехваченный `fetch`); он покрывает валидацию конфигурации, маршрутизацию, разбор возможностей, каждую стратегию, шлюз инструментов, шлюз изображений и smart-маршрутизацию. Живой smoke-тест — единственный, который касается сети, и только при наличии ключа.

## Известные ограничения / ещё не подключено

- **Нативный NDJSON-стриминг изображений отложен.** Зрение работает на openai-совместимом пути; стримовый запрос с изображением, попадающий на нативный бэкенд `/api/chat` (`api_mode: native` + изображения + `stream: true`), возвращает чистый `501`, а не полуподключённый стрим. Нестримовое нативное зрение работает.
- **Ещё нет round-robin, семантического кеша и маршрутизации по размеру контекста** — это Фаза 6 / будущее. (Решения smart-маршрутизатора *кешируются* для идентичных тел запросов, а одновременные идентичные запросы объединяются — см. `smart` выше, — но семантического/векторного кеша и маршрутизации по количеству токенов нет.)
- **BinEval только отчитывается.** Низкий `X-Fusion-Bineval-Score` не запускает повторное обдумывание или повторный синтез — прокси показывает балл (или `X-Fusion-Bineval-Skipped`), а решение за клиентом. Опциональный авто-повтор рассматривали и осознанно не подключили (умножение стоимости на самых дорогих ходах, ложноотрицательные LLM-судьи и дрейф модели-оценщика как тихий усилитель регрессий).
- **Только loopback.** По умолчанию слушает `127.0.0.1`, без TLS, один пользователь (опциональный общий клиентский токен, а не аутентификация отдельных пользователей). Не для многопользовательского выставления наружу.
- **Failover + стриминг**: цепочка может переключиться только *до* первого токена; сбой upstream посреди стрима всплывает как ошибка стрима (частично отправленный ответ тихо перезапустить нельзя).

## Для AI-агентов

Вы — AI-агент, которому нужно подключить себя (или обёртку, в которой вы работаете) к llm-fusion. Этот раздел самодостаточен: для рабочей настройки больше ничего в этом файле читать не нужно.

### 1. Проверьте, работает ли прокси

```bash
curl -sf http://127.0.0.1:8080/health   # 200 → процесс жив
curl -sf http://127.0.0.1:8080/ready    # 200 → upstream достижим; 503 → проблема с upstream
```

Если не работает — запустите из корня репозитория:

```bash
cd <путь-к-llm-fusion> && npm start
```

Для старта нужен файл `.env` с `OLLAMA_API_KEY=<ключ>`. Если ключа нет, попросите его у человека-оператора — никогда не выдумывайте ключ и не переносите его из другого места.

### 2. Выберите точку входа под свой диалект API

- **Диалект OpenAI Chat Completions** → базовый адрес `http://127.0.0.1:8080/v1`, запросы на `POST /v1/chat/completions`.
- **Диалект Anthropic Messages** (Claude Code и похожие) → базовый адрес — **корень** прокси `http://127.0.0.1:8080` (не `/v1`), запросы на `POST /v1/messages`.
- Доступные виртуальные имена моделей узнавайте во время работы из `GET /v1/models` — это лучше жёсткого кодирования, потому что `fusion.yaml` перезагружается горячо и список моделей может измениться, пока прокси работает.

### 3. Аутентификация

- По умолчанию прокси без аутентификации на localhost. Отправляйте любую непустую строку как ключ API (например `local-no-auth`) — некоторые клиентские SDK отказываются работать с пустым ключом.
- Если оператор включил аутентификацию клиентов (`FUSION_PROXY_TOKEN` + `server.auth_token_env` в `fusion.yaml`): отправляйте `Authorization: Bearer <токен>`. Ответ `401` означает, что токен нужен или неверен.

### 4. Выберите модель под задачу

| Модель | Для чего |
|---|---|
| `fusion-agents` | **По умолчанию для автономных циклов.** Умная маршрутизация: рутинные шаги стоят 1 upstream-вызов; сложные шаги и восстановление после ошибок автоматически получают полную панель. |
| `fusion-coder` | Ходы с кодом и планированием, где панель нужна всегда. |
| `fusion-researcher` | Исследования, анализ, написание отчётов. |

### 5. Контракт вызовов инструментов (на что можно положиться)

- Прокси выдаёт **ровно один `tool_calls` на шаг** независимо от стратегии — панель и судья работают целиком на стороне сервера и никогда не видят вашу реальную схему `tools`, так что ваш существующий цикл инструментов работает без изменений, а побочные эффекты остаются ровно-один-раз.
- Когда последний результат инструмента выглядит как сбой (error / exception / ненулевой код выхода), `fusion-agents` эскалирует этот ход прямо в полный fusion — просить более глубоких рассуждений самостоятельно не нужно.
- Вызов инструмента, обрезанный лимитом токенов, сообщается честно (`stop_reason: "max_tokens"` на диалекте Anthropic), а не как якобы пригодный к исполнению сломанный JSON.

### 6. Готовые скрипты запуска (вместо ручного подключения)

```bash
./bin/fusion-opencode <модель>    # поднимет прокси при необходимости, запишет конфигурацию провайдера OpenCode, откроет OpenCode
./bin/fusion-claude <модель>      # поднимет прокси при необходимости, экспортирует переменные ANTHROPIC_*, откроет Claude Code
```

Оба учитывают `FUSION_PROXY_URL` (по умолчанию `http://127.0.0.1:8080`) и `FUSION_PROXY_TOKEN`. Оба принимают `run "<запрос>"` после имени модели для разового запуска без интерфейса.

### 7. Минимальная проверка (скопируйте и запустите)

```bash
curl -s http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-agents","messages":[{"role":"user","content":"Reply with the single word: ok"}]}'
```

JSON-ответ означает, что всё настроено. Детальное подключение Claude Code — в [`docs/claude-code.md`](./docs/claude-code.md); каждый ключ конфигурации задокументирован в [`fusion.example.yaml`](./fusion.example.yaml).
