# llm-fusion — Fusion Proxy

> [English](./README.md) | **Русский** | [Українська](./README.ua.md)
>
> Перевод соответствует релизу v0.1.19. Каноническая версия — [английская](./README.md).

> **Самостоятельно разворачиваемый прокси коллегиального рассуждения для Ollama Cloud, совместимый с OpenAI API и Anthropic Messages API.**  
> Одно виртуальное имя модели запускает панель моделей, судью и синтезатор — или умный маршрутизатор, который для каждого запроса решает, стоит ли дело такой тяжёлой обработки вообще.

Если вы знаете **OpenRouter Fusion**, идею вы уже уловили: *думают многие модели — отвечает одна*.  
**llm-fusion — тот же подход, перестроенный для разработчиков, которым нужны контроль, прозрачность и безопасность агентных циклов.** Он работает локально, маршрутизирует запросы в Ollama Cloud и добавляет стратегию `smart`, которую OpenRouter оставляет на откуп клиенту.

Без базы данных. Без шага сборки. Node 24 + TypeScript + Hono, `.ts` исполняется напрямую через `tsx`. Одна конфигурация YAML.

## Быстрый старт

```bash
git clone https://github.com/Lexus2016/LLM-Fusion.git llm-fusion
cd llm-fusion
npm install                                        # Node ≥ 24
printf 'OLLAMA_API_KEY=ollama-your-key\n' > .env    # ваш ключ Ollama Cloud
npm start                                          # прокси на http://127.0.0.1:8080
```

Дальше направьте любой OpenAI-совместимый клиент на `http://127.0.0.1:8080/v1` и запрашивайте модель по имени — или запустите OpenCode одной командой:

```bash
./bin/fusion-opencode fusion-coder                # поднимет прокси при надобности, подключит OpenCode, откроет TUI
```

Впервые здесь? Сначала прочитайте **[Какую модель выбрать?](#какую-модель-выбрать)**.

## Почему llm-fusion, а не OpenRouter Fusion?

Оба прогоняют запрос через панель моделей плюс шаг судьи/синтезатора, чтобы получить более сильный ответ. Разница — в том, **где, когда и как** работает этот конвейер.

| | **llm-fusion** | **OpenRouter Fusion** |
|---|---|---|
| **Развёртывание и контроль** | Работает на вашей машине / в Docker. Один пользователь, полная прозрачность, один файл конфигурации. | Полностью управляемый SaaS на OpenRouter. |
| **Счёт** | Платите Ollama Cloud напрямую. | Платите OpenRouter; использование агрегировано. |
| **Охват провайдеров** | Только Ollama Cloud (один счёт, один upstream). | Любой провайдер на OpenRouter (более широкий каталог моделей). |
| **Автоматическая маршрутизация** | Встроенная стратегия `smart`: быстрый LLM-маршрутизатор для **каждого запроса** выбирает `single` (дёшево) или `fusion` (глубоко). | Fusion всегда запускает полную панель; маршрутизация — вручную или отдельным маршрутизатором. |
| **Безопасность агентных циклов** | Выдаёт **ровно один `tool_calls`** на шаг; панель в режиме `deliberate` никогда не касается настоящих инструментов. | Плагин возвращает анализ; финальный вызов инструмента решает вызывающая модель. |
| **Рычаг стоимости для длинных сессий** | `fusion_planning_turn_only`: полная панель на ходе планирования, дальше только синтезатор (5 вызовов → 1) для механических шагов внутри цикла. | Нет автоматического удешевления отдельных запросов внутри цикла. |
| **Веб-заземление** | Опциональное, по явному включению: один поиск Tavily перед панелью, результаты вставляются текстовым контекстом. Условия — `TAVILY_API_KEY` (окружение) **и** `web_search.enabled` (конфигурация); без обоих полностью ВЫКЛЮЧЕНО. | Встроенные `web_search` + `web_fetch` для панели и судьи. |
| **Устойчивость** | Цепочки `failover` + плавная деградация + предел `max_concurrency` с помодельными бюджетами. | Управляемая надёжность + защита от рекурсии. |
| **Стиль конфигурации** | Один YAML с горячей перезагрузкой и эмпирически подобранными пресетами. | Plugin JSON / серверный инструмент / слаг модели; пресеты + веб-интерфейс. |

**Выбирайте llm-fusion, когда:**

- Ваш автономный агент работает **сотни шагов** и требует детерминированных побочных эффектов ровно один раз.
- Вы хотите, чтобы рутинные шаги стоили **1 вызов к провайдеру**, а fusion оставался для сложных шагов и восстановления после ошибок.
- Вам нужен **локальный, прозрачный процесс** с одним файлом конфигурации и без дополнительной привязки к поставщику.
- Ваши модели и бюджет живут на **Ollama Cloud**.

**Выбирайте OpenRouter Fusion, когда:**

- Нужны **исследования с веб-заземлением** на передовых моделях многих провайдеров.
- Хочется **управляемого сервиса** с веб-площадкой и глобальной инфраструктурой.
- Вы делаете разовое глубокое исследование, а не длинные агентные циклы.

**Суть:** OpenRouter Fusion оптимизирует потолок качества одиночного ответа в исследовательских задачах. **llm-fusion оптимизирован для работы внутри агентного цикла** — контролирует стоимость, не допускает двойных вызовов инструментов и включает тяжёлое рассуждение только там, где оно действительно окупается.

## Какую модель выбрать?

В `fusion.yaml` поставляются три **специализированных под задачи** пресета, собранных по результатам эмпирического отбора моделей (8 моделей × 3 пробы — программирование, исследования, вызов инструментов):

| Вызывайте эту модель | Для чего | Стратегия | Как собрана |
|---|---|---|---|
| **`fusion-coder`** | программирование, планирование, аудит кода | `fusion` | панель `glm-5.2` + `kimi-k2.7-code` + `gpt-oss:120b` → судья `glm-5.2` → синтезатор `kimi-k2.7-code` |
| **`fusion-researcher`** | исследования, анализ, отчёты | `fusion` | панель `kimi-k2.7-code` + `glm-5.2` + `gpt-oss:120b` → судья `glm-5.2` → синтезатор `kimi-k2.7-code` |
| **`fusion-agents`** | автономные агентные циклы | `smart` | маршрутизатор `glm-5.2`; простые шаги → `glm-5.2`, сложные шаги и восстановление после ошибок → панель `fusion-coder` |

Два правила вытекли прямо из данных:

- **Для программирования — fusion, а не одна модель.** Архитектура и планирование действительно выигрывают от нескольких точек зрения. (Чистый *аудит* кода — простое перечисление проблем — единственная «программистская» задача, где побеждает одна модель, и она не репрезентативна для программирования в целом.)
- **Панели намеренно смешивают линии моделей.** `glm`/`kimi`/`deepseek`/`minimax`/`qwen` — всё китайские лаборатории с общими слепыми зонами; каждая панель добавляет западный декоррелятор (`gpt-oss:120b` — линия OpenAI), чтобы ошибки панели коррелировали меньше — в этом и смысл панели. (Изначальным декоррелятором был Gemini, но он отклоняет историю вызовов инструментов, созданную другими моделями, с ошибкой `thought_signature` на каждом шаге внутри цикла, поэтому его заменили.)

Исходные универсальные пресеты (`fusion-1`, `smart-1`, `fast-glm` / `fast-kimi` / `fast-deepseek`) тоже остаются — для разовых нужд.

## Архитектура

Запрос к `POST /v1/chat/completions` несёт **виртуальное имя модели**. Прокси сопоставляет это имя с одной из четырёх стратегий и исполняет:

| Стратегия | Что делает |
|----------|--------------|
| `single` | Прямая передача 1:1 к одной модели провайдера (поток и без потока). Примитив, на котором построено всё остальное. |
| `failover` | Упорядоченная цепочка; при сбое до первого токена переходит к следующему участнику. Устойчивость без разветвления. |
| `fusion` | Конвейер рассуждения: **панель → судья → синтезатор** (ниже). |
| `smart` | LLM-**маршрутизатор** классифицирует каждый запрос и направляет в `single` (дёшево) или `fusion` (глубоко). |

### Fusion: панель → судья → синтезатор

1. **Панель** — запрос идёт к N моделям *параллельно* (по умолчанию 3). В режиме инструментов `deliberate` участники панели получают *описания* инструментов текстом, но не настоящую схему `tools`, поэтому ниже по конвейеру всегда появляется лишь один канонический вызов инструмента. Опционально один участник работает **состязательным рецензентом** (мандат «красной команды»: найди изъян, отстаивай противоположную позицию) — см. «Известные ограничения». Длинные контексты сжимаются до разветвления панели, чтобы не переполнять контекстные окна участников.
2. **Судья** — один вызов со структурированным JSON сравнивает ответы панели (консенсус, расхождения, уникальные наблюдения, слепые зоны, частичное покрытие, признаки галлюцинаций) и выдаёт **калиброванную уверенность** (`high`/`medium`/`low`) вместе с `fragile_claims`; синтезатор смягчает шаткие и малоуверенные утверждения вместо того, чтобы выдавать общие априорные представления за твёрдую определённость. Если судья возвращает некорректный JSON, этап плавно деградирует к сырым ответам панели, а не завершается ошибкой.
3. **Синтезатор** — финальная модель пишет ответ, который получает клиент, стримит по запросу и **единственная** получает настоящую схему `tools` (поэтому до агента доходит ровно один `tool_calls`). Опционально после него — оценка фактической согласованности **BinEval**, возвращаемая в заголовках.

### Smart: маршрутизатор

`smart` делает **один** непотоковый вызов с `temperature: 0` и только-JSON ответом к быстрой модели-`router`, читает `{ "route": "simple" | "fusion" }` и направляет в соответствующую подстратегию. Любая ошибка маршрутизатора, тайм-аут или неразборчивый ответ ведут к настроенному маршруту `default` (по умолчанию `simple`, потому что контроль стоимости — главная цель). Подстратегии задаются встроенными блоками или строковыми ссылками на другие настроенные модели. Решения маршрутизатора кешируются по идентичному телу запроса, а одновременные идентичные запросы объединяются, поэтому всплеск одинаковых ходов повторно использует один вызов маршрутизатора. Если последний результат инструмента похож на сбой (ошибка / исключение / ненулевой код выхода), запрос идёт сразу в `fusion` мимо маршрутизатора — восстановление и есть то место, где рассуждение окупается.

### Опциональные возможности fusion

Все они **по умолчанию ВЫКЛЮЧЕНЫ**; включаются отдельно для каждой fusion-модели в `fusion.yaml`. Полный справочник: [`fusion.example.yaml`](./fusion.example.yaml).

- **Калиброванная уверенность судьи** — судья выдаёт `confidence` (`high`/`medium`/`low`) и `fragile_claims` (спорные / подтверждённые одним источником / слабые утверждения). Синтезатор смягчает такие утверждения. Судье **предписано**, что любые `hallucination_flags` или `fragile_claims` должны снижать `confidence` до `medium`/`low` (это правило на уровне подсказки, а не жёсткая проверка в коде — схема принимает JSON судьи как есть).
- **Состязательный участник панели** — `adversarial: <участник>` превращает одно место в рецензента «красной команды»: отстаивать противоположную позицию, искать изъяны, скрытые допущения, краевые случаи и состояния гонки вместо согласия с консенсусом. Ролевая декорреляция поверх декорреляции по линиям моделей. Состязательного участника **никогда не отменяют досрочно**, панель его дожидается, поэтому его ответ не теряется; ему предписано говорить «настоящей проблемы не вижу», когда консенсус прочен. Проверяется, что это существующий участник панели.
- **Веб-заземление (Tavily)** — `web_search: { enabled: true, max_results, timeout_s, max_context_chars, max_prompt_chars }`. Прокси выполняет **один** поиск Tavily до разветвления панели и вставляет очищенные результаты текстовым контекстом каждому участнику (никто не получает настоящие инструменты, поэтому инвариант одного `tool_calls` сохраняется). Условия — `TAVILY_API_KEY` в окружении *и* `web_search.enabled` на модели (нет ключа или включения → нет поиска, задержки и затрат), **плюс два пропуска во время исполнения**: если в последнем сообщении пользователя нет пригодного текста запроса или подсказка уже превышает `max_prompt_chars` (по умолчанию 80 000) — размерный предохранитель не даёт веб-контексту переполнить участника с меньшим окном (например, gpt-oss:120b со 128k) в длинных продолжениях агентного цикла. Контекст вставляется ходом `user` (не `system`) с датой в начале, чтобы модели с устаревшей границей знаний (kimi-k2.7-code) использовали свежие факты, а не отказывались. Неудачный поиск молча деградирует к панели без заземления. Уважает `fusion_planning_turn_only`. Ключ добавьте в `.env`:
  ```bash
  echo 'TAVILY_API_KEY=tvly-...' >> .env
  ```
- **Пост-синтезаторная проверка BinEval** — `bineval: { enabled: true, model: <eval>, threshold: 0.7, dimensions: [...] }`. После успешного *непотокового* синтезатора один дополнительный вызов оценщика выставляет баллы за фактическую согласованность (или по вашим бинарным вопросам) и возвращает результат в заголовках ответа: `X-Fusion-Bineval-Score` (0–1), `X-Fusion-Bineval-Dimensions` (JSON по измерениям) и `X-Fusion-Bineval-Low-Score: true`, когда общий балл ниже `threshold`. Если BinEval настроен, но оценка не состоялась, прокси ставит `X-Fusion-Bineval-Skipped: <причина>`, чтобы клиент отличал «балл высокий» от «оценка не запускалась» — причины: `streaming`, `synth_error` (синтезатор ≥400), `eval_failed` (оценщик упал/превысил время/разомкнута цепь), `empty_output` (ответ только с инструментами), `non_json_body`, `synth_only` (путь только-синтезатор внутри цикла / обход). **BinEval только отчитывается** — он не запускает повторную маршрутизацию или повторное рассуждение.

### Надёжность агентных циклов (v0.1.16+)

Длинные агентные сессии раньше гибли на четырёх зависимых от нагрузки режимах отказа; теперь все четыре закрыты структурно:

- **Защита полноты синтезатора с переходом на судью.** «Мыслящая» модель-синтезатор может объявить себя завершённой, ещё находясь посреди плана (пустой ответ или один встроенный блок `<think>` без артефакта). Защита обнаруживает это и на потоковом, и на непотоковом пути, повторяет того же синтезатора один раз со строгой подсказкой завершения, а если и повтор непригоден (или сам был обрезан лимитом токенов посреди вызова инструмента) — делает ОДНУ попытку моделью-судьёй, так что ни одна отдельная модель не может остановить агентный цикл. Во время тихого восстановления поток несёт SSE-комментарии `: keepalive` (каждые 5 с, настраивается через `FUSION_SYNTH_RECOVERY_PING_MS`), чтобы клиенты и промежуточные узлы не разрывали соединение по тайм-ауту.
- **Честный `stop_reason` при обрезке лимитом токенов.** Вызов инструмента, обрезанный `max_tokens`, сообщается Anthropic-клиентам как `stop_reason: "max_tokens"` — а не как исполняемый `tool_use` с битым JSON на входе. Если же лимит сработал ровно *после* завершения JSON вызова, вызов остаётся исполняемым (`tool_use`) — готовый многоминутный Write никогда не выбрасывается.
- **Помодельные бюджеты конкурентности.** `upstream.per_model_concurrency` даёт каждой реальной модели провайдера собственные ворота перед глобальным лимитом — всплеск фоновых вызовов мелкой модели (Claude Code выдаёт их по 80–130 в минуту) встаёт в очередь у собственных ворот вместо блокировки интерактивных ходов в голове общей очереди.
- **Разделённые классы трафика.** Лаунчер `fusion-claude` по умолчанию направляет фоновую модель Claude Code (`ANTHROPIC_SMALL_FAST_MODEL`) на `fast-deepseek` — модель, от которой не зависит ни одна панель, судья, синтезатор, маршрутизатор или simple-маршрут, — поэтому фоновые всплески не могут исчерпать лимит модели, которая пишет ваши файлы.

## Честная заметка о стоимости (прочитайте)

Полный fusion выполняется на **каждом** шаге. Агентный цикл (чтение → размышление → правка → тесты → повторное чтение…) умножает **вызовы API моделей** к провайдеру:

```
вызовов_на_шаг = N_панели + 1 (судья) + 1 (синтезатор) = 3 + 1 + 1 = 5
```

Типичная программистская задача на **15–25 шагов**, следовательно, выдаёт примерно **75–125 вызовов моделей**, если fusion включён всегда.

- **Исполнение инструментов НЕ умножается** — инструменты запускаются раз на шаг (канонический `tool_calls` выдаёт только синтезатор). Действия вашего агента не затронуты.
- **Вызовы API моделей УМНОЖАЮТСЯ** — на множитель «панель + судья + синтезатор». Это и есть цена, принятая для v1.

**Смягчение (уже в коробке, не в планах): стратегия `smart`.** Направьте агента на `fusion-agents` — и рутинные шаги (`read_file`, `grep`, …) пойдут дешёвым путём `single` (1 вызов), а действительно сложные получат полный fusion. Другие встроенные рычаги:

- маленькая панель по умолчанию (3),
- глобальный предел `max_concurrency` плюс **помодельные бюджеты конкурентности** (`per_model_concurrency`) — всплеск на одной модели встаёт в очередь у собственных ворот, не блокируя интерактивные ходы,
- сжатые тайм-ауты панели/судьи (90 с / 120 с в пресетах, оба под потолком провайдера ~182 с),
- переключатель `fusion_planning_turn_only` (полная панель на каждом ходе планирования — то есть на каждом запросе, где последнее сообщение является свежей инструкцией пользователя, — и деградация до только-синтезатора — 5 вызовов → 1 — лишь на продолжениях с результатом инструмента внутри цикла; новая задача глубоко в длинной сессии всё равно получит панель).

Именно так llm-fusion держит длинные агентные циклы доступными по цене, не жертвуя рассуждением там, где оно важно.

## Обновление

`llm-fusion` живёт в git и не публикуется в npm, поэтому «обновиться» — это подтянуть последнее и освежить зависимости:

```bash
git pull                          # перемотка к новейшему коммиту / тегу
npm install                       # подхватить изменения зависимостей
```

Затем перезапустите прокси. Ваши `fusion.yaml` и `.env` — ваши: `git pull` никогда не перезаписывает локальные изменения в них (поставляемый `fusion.yaml` содержит пресеты, которые вы могли редактировать; если редактировали — закоммитьте/спрячьте изменения или держите свою копию, указанную через `FUSION_CONFIG`). Чтобы зафиксироваться на конкретном релизе вместо ветки `main`:


`fusion.example.yaml` — полностью аннотированный справочник, который обновляется между релизами: сравнивайте его со своей рабочей конфигурацией, когда выходит новая версия, чтобы увидеть новые опции. Журнал изменений: [`CHANGELOG.md`](./CHANGELOG.md), релизы: <https://github.com/Lexus2016/LLM-Fusion/releases>.

## Настройка

По умолчанию прокси загружает `./fusion.yaml` (путь переопределяется через `FUSION_CONFIG`). Полная рабочая конфигурация:

```yaml
upstream:
  base_url: https://ollama.com
  api_key_env: OLLAMA_API_KEY      # ИМЯ переменной окружения с ключом — никогда не сам ключ
server:
  bind: 127.0.0.1
  port: 8080
models:
  fast-glm:                        # single — прямая передача 1:1
    strategy: single
    target: glm-5.2
  fusion-1:                        # fusion — панель -> судья -> синтезатор
    strategy: fusion
    panel: [glm-5.2, kimi-k2.7-code, deepseek-v4-pro]
    judge: glm-5.2
    synth: deepseek-v4-pro
  smart-1:                         # smart — маршрутизатор выбирает single или fusion
    strategy: smart
    router: glm-5.2
    default: simple
    simple: fast-glm               # строковая ссылка на single-модель выше
    fusion: fusion-1               # строковая ссылка на fusion-модель выше
```

Это базовые примитивы. **Поставляемый `fusion.yaml`** также определяет три специализированных пресета из раздела [Какую модель выбрать?](#какую-модель-выбрать) — `fusion-coder`, `fusion-researcher`, `fusion-agents`. Всё незаданное получает задокументированные значения по умолчанию. **Полностью аннотированный справочник** — каждый ключ, каждый дефолт, `failover`, `tool_mode`, `fusion_planning_turn_only`, `overrides`, встроенные smart-блоки — см. [`fusion.example.yaml`](./fusion.example.yaml).

Шпаргалка стратегий для карты `models:`:

- **`single`** → `{ strategy: single, target: <модель> }`
- **`failover`** → `{ strategy: failover, chain: [<m1>, <m2>, …] }`
- **`fusion`** → `{ strategy: fusion, panel: [...], judge: <модель>, synth: <модель> }`
- **`smart`** → `{ strategy: smart, router: <модель>, default: simple|fusion, simple: <блок-или-ссылка>, fusion: <блок-или-ссылка> }`

Конфигурация **перезагружается горячо**: редактируйте `fusion.yaml` — изменения маршрутизации и моделей применяются вживую (некорректная правка отклоняется, предыдущая конфигурация сохраняется). Изменения блока `upstream` — базовый адрес, переменная ключа, конкурентность — требуют перезапуска.

## Переменные окружения

Прокси читает обычные переменные окружения. Также он **автоматически загружает локальный `.env`** на старте (нативно в Node 24; отсутствие файла не ошибка).

| Переменная | Обязательна | По умолчанию | Назначение |
|----------|----------|---------|---------|
| `OLLAMA_API_KEY` | Да, для живой работы | — | Bearer-ключ Ollama Cloud. Имя — то, на что указывает `upstream.api_key_env` (по умолчанию `OLLAMA_API_KEY`). Держится только на сервере; клиентам не отправляется и не логируется. |
| `FUSION_PROXY_TOKEN` | Нет | не задано | Если в конфигурации есть `server.auth_token_env: FUSION_PROXY_TOKEN`, клиенты обязаны слать `Authorization: Bearer <это значение>`. Не задано ⇒ прокси без аутентификации (localhost, один пользователь) и предупреждает на старте. |
| `FUSION_CONFIG` | Нет | `./fusion.yaml` | Путь к файлу конфигурации. |
| `LOG_PRETTY` | Нет | не задано | `LOG_PRETTY=1` включает человекочитаемые логи (иначе — структурированный JSON). |
| `LOG_LEVEL` | Нет | `info` | Уровень логов pino (`debug`, `info`, `warn`, …). |
| `FUSION_BIND` | Нет | `server.bind` | Переопределяет адрес прослушивания без правки конфигурации (используется Docker-образом). |
| `FUSION_SYNTH_RECOVERY_PING_MS` | Нет | `5000` | Интервал SSE-комментариев `: keepalive`, идущих в поток, пока внутри него выполняется повтор восстановления синтезатора. Положительное число; всё прочее возвращает значение по умолчанию. |

`.env.example` здесь закоммитить нельзя (песочница блокирует). Создайте собственный `.env` в корне проекта с таким содержимым:

```dotenv
# Обязательно для любой живой (нетестовой) работы:
OLLAMA_API_KEY=ollama-your-key-here

# Опционально — раскомментируйте, чтобы требовать клиентский токен (также задайте
# server.auth_token_env: FUSION_PROXY_TOKEN в fusion.yaml):
# FUSION_PROXY_TOKEN=choose-a-long-shared-secret

# Опциональные переопределения:
# FUSION_CONFIG=./fusion.yaml
# LOG_PRETTY=1
```

## Интеграция с OpenCode

Направьте OpenCode на прокси блоком провайдера в `opencode.json`:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "fusion": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Fusion Proxy",
      "options": { "baseURL": "http://127.0.0.1:8080/v1" },
      "models": {
        "fusion-coder": { "name": "Fusion Coder" },
        "fusion-researcher": { "name": "Fusion Researcher" },
        "fusion-agents": { "name": "Fusion Agents" },
        "fusion-1": { "name": "Fusion 1" },
        "smart-1": { "name": "Smart 1" }
      }
    }
  }
}
```

Идентификаторы моделей должны совпадать с виртуальными именами в вашем `fusion.yaml`. Если задан `FUSION_PROXY_TOKEN` (клиентская аутентификация), добавьте токен для провайдера `fusion` через `opencode auth login` (или ваш `auth.json`), чтобы OpenCode слал его Bearer-токеном.

Для автономного агентного цикла правильный выбор по умолчанию — **`fusion-agents`**: рутинные шаги остаются дешёвыми (маршрутизатор → быстрая одиночная модель), а панель fusion включается на сложных шагах и восстановлении после ошибок. Для агента-программиста — **`fusion-coder`**; для исследований — **`fusion-researcher`**.

### Запуск OpenCode с fusion-моделью

С запущенным прокси (`npm start`) и блоком провайдера выше в вашем `opencode.json`
(локальном для проекта или глобальном `~/.config/opencode/opencode.json`) модель выбирается через `-m fusion/<имя>`:

```bash
opencode -m fusion/fusion-coder                         # интерактивный TUI, программирование / планирование
opencode -m fusion/fusion-agents                        # интерактивный TUI, автономная маршрутизация
opencode run -m fusion/fusion-researcher "Summarize X"  # headless, один запрос
opencode run -m fusion/fusion-agents "Fix the bug in Y" # headless
```

### Лаунчер одной командой (`fusion-opencode`)

Сам `ollama` расширить нельзя, но репозиторий поставляет собственный лаунчер, чтобы вам не приходилось
вручную поднимать прокси или редактировать конфигурации. Он (1) запускает прокси, если тот ещё не работает,
(2) прописывает провайдера `fusion` в вашу глобальную конфигурацию OpenCode — идемпотентно, трогает
только ключ `fusion` — и (3) запускает OpenCode с выбранной моделью:

```bash
./bin/fusion-opencode fusion-coder             # TUI для программирования / планирования
./bin/fusion-opencode fusion-researcher        # TUI для исследований / отчётов
./bin/fusion-opencode fusion-agents            # TUI для автономных агентных циклов
./bin/fusion-opencode fusion-agents run "Fix the failing test in src/foo.ts"   # headless разово
```

Добавьте его в `PATH`, чтобы вызывать `fusion-opencode <модель>` из любой папки:

```bash
npm link            # или: ln -s "$PWD/bin/fusion-opencode" ~/.local/bin/fusion-opencode
fusion-opencode fusion-coder
```

Уважает `FUSION_PROXY_URL` (по умолчанию `http://127.0.0.1:8080`) и, если прокси требует клиентскую
аутентификацию, `FUSION_PROXY_TOKEN` (используется как apiKey, который шлёт OpenCode).

### Интеграция с Claude Code

Claude Code говорит на **Anthropic Messages API**. llm-fusion открывает `POST /v1/messages` на том же
базовом адресе, поэтому Claude Code можно направить на корень прокси несколькими переменными окружения.

#### Лаунчер одной командой (`fusion-claude`)

`bin/fusion-claude` запускает прокси (при надобности), экспортирует переменные окружения Anthropic и открывает
Claude Code с выбранной моделью:

```bash
./bin/fusion-claude fusion-agents            # автономные агентные циклы (по умолчанию)
./bin/fusion-claude fusion-coder             # программирование / планирование
./bin/fusion-claude fusion-researcher        # исследования / отчёты
./bin/fusion-claude fusion-agents run "Fix the failing test in src/foo.ts"
```

Он уважает `FUSION_PROXY_URL` и `FUSION_PROXY_TOKEN` так же, как `fusion-opencode`. Ключевое
отличие от OpenAI-совместимых клиентов — базовый адрес: Claude Code использует **корень** прокси
(`http://127.0.0.1:8080`), а не `/v1`, потому что вызывает `/v1/messages`.

Полное руководство по настройке, справочник переменных окружения и решение проблем:
[`docs/claude-code.md`](./docs/claude-code.md).

### Для AI-агентов (любой OpenAI-совместимый клиент)

Прокси — готовая замена эндпоинта OpenAI Chat Completions, поэтому работает любой агентный фреймворк, говорящий на OpenAI, — не только OpenCode (Continue, Cline, Aider с базовым адресом OpenAI, ваш собственный цикл). Подключение:

- **Базовый адрес:** `http://127.0.0.1:8080/v1`
- **Ключ API:** любая непустая строка, пока прокси без аутентификации (например, `local-no-auth`); настоящее значение `FUSION_PROXY_TOKEN`, если вы включили клиентскую аутентификацию.
- **Модель:** по задаче — `fusion-coder`, `fusion-researcher` или `fusion-agents` (см. [Какую модель выбрать?](#какую-модель-выбрать)).

Прокси выдаёт **ровно один** `tool_calls` на шаг независимо от стратегии, поэтому существующий цикл инструментов агента работает без изменений — панель и судья fusion происходят целиком на сервере и невидимы для агента. Для длинных автономных сессий берите `fusion-agents`: это `smart`, поэтому полная панель тратится только там, где окупается, иначе — быстрая одиночная модель, а при ошибочном результате инструмента рассуждение включается автоматически (`escalate_on_tool_error`). Минимальный прямой вызов:

```bash
curl http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-agents","messages":[{"role":"user","content":"List the open TODOs in this repo"}]}'
```

## Эндпоинты

| Метод и путь | Назначение |
|---------------|---------|
| `POST /v1/chat/completions` | Главная точка вывода. OpenAI-совместимая; поддерживает `stream`, `tools`, `tool_choice` и блоки изображений. Маршрутизируется виртуальным именем `model`. |
| `POST /v1/messages` | Точка Anthropic Messages API для Claude Code. Переводит блоки содержимого Anthropic во внутренний OpenAI-конвейер и обратно; поддерживает стриминг и инструменты. |
| `GET /v1/models` | Перечень настроенных виртуальных моделей (формат списка OpenAI). Добавляет `context_window` / `supports_vision`, где известны возможности. |
| `GET /health` | Живучесть. `200`, если процесс работает. Без проверки провайдера. |
| `GET /ready` | Готовность. `200` только когда провайдер достижим и репрезентативная модель обнаруживается; иначе `503`. |

## Фаза 0 — живая верификация

Корректность адаптера провайдера держится на допущениях об Ollama Cloud (Bearer + SSE, вызов инструментов по моделям, обнаружение через `/api/show`, формат изображений). Проверьте их на живом API своим ключом:

```bash
OLLAMA_API_KEY=ollama-... npm run smoke
```

Это исполняет `test/live.smoke.test.ts` против `https://ollama.com`. Каждая проверка — отдельный тест с ясным результатом:

- **A-1 — Bearer + завершение** *(ПРОВЕРЕНО 2026-06-26)*: непотоковый чат к `glm-5.2` возвращает ответ (подтверждает ключ и openai-совместимый путь).
- **SSE — стриминг** *(ПРОВЕРЕНО 2026-06-26)*: тот же запрос со `stream: true` возвращает SSE-фрагменты `data:`.
- **A-2 — инструменты по моделям** *(ПРОВЕРЕНО 2026-06-26 — `tool_calls` на всех: `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`)*: запрос с `tools` к каждой из трёх моделей; логирует **PASS/FAIL** по модели (набор не падает, если модель без инструментов, — но синтезатором должна быть модель со статусом PASS, потому что весь deliberate-конвейер зависит от её `tool_calls`).
- **A-3 — `/api/show`** *(ПРОВЕРЕНО 2026-06-26 — `capabilities[]` + `*.context_length` для всех трёх)*: подтверждает, что для этих моделей возвращаются `capabilities[]` и `*.context_length`, и логирует найденное.
- **A-4/A-5 — изображения** *(ЕЩЁ ОТКРЫТО — перепроверить на `kimi-k2.7-code`; прежняя тестовая модель `qwen3-vl:235b` изъята 2026-06-16)*: отправляет PNG 1×1 через `image_url` модели со зрением (`kimi-k2.7-code` по умолчанию; переопределение — `FUSION_VISION_MODEL`) и логирует, принят ли формат изображений OpenAI.

В конце — сводка, по которой вы фиксируете две настройки в `fusion.yaml`: `upstream.api_mode` (`openai`, если проверка изображений приняла формат OpenAI, иначе `native`) и какую модель безопасно ставить fusion-синтезатором (ту, чья строка A-2 — PASS).

Без `OLLAMA_API_KEY` набор **пропускается**, поэтому он никогда не исполняется в CI или в типовом прогоне тестов.

## Известные ограничения / ещё не подключено

- **Нативный NDJSON-стриминг изображений отложен.** Зрение работает на openai-совместимом пути; потоковый запрос с изображением, попадающий на нативный бэкенд `/api/chat` (`api_mode: native` + изображения + `stream: true`), возвращает чистый `501`, а не полуподключённый поток. Непотоковое нативное зрение работает.
- **Пока нет циклического распределения, семантического кеша и маршрутизации по размеру контекста** — это Фаза 6 / будущее. (Решения smart-маршрутизатора *кешируются* по идентичному телу запроса, одновременные идентичные запросы объединяются — см. `smart` выше, — но нет семантического/векторного кеша и маршрутизации по числу токенов подсказки.)
- **BinEval только отчитывается.** Низкий `X-Fusion-Bineval-Score` не запускает повторное рассуждение или повторный синтез — прокси показывает балл (или `X-Fusion-Bineval-Skipped`), а решение принимает клиент. Автоматический повтор рассматривался и намеренно не подключён (усиление стоимости на самых дорогих ходах, ложные срабатывания LLM-судьи и дрейф модели-оценщика как тихий усилитель регрессий).
- **Только локальная петля.** По умолчанию слушает `127.0.0.1`, без TLS, один пользователь (опциональный общий клиентский токен, а не аутентификация отдельных пользователей). Не рассчитано на многопользовательское открытие наружу.
- **Failover + стриминг**: цепочка может переключиться только *до* первого токена; сбой провайдера посреди потока становится ошибкой потока (частично отправленный ответ нельзя тихо перезапустить).
