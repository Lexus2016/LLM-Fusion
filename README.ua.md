# llm-fusion — Fusion Proxy

> [English](./README.md) | [Русский](./README.ru.md) | **Українська**
>
> Переклад відповідає релізу v0.1.19. Канонічна версія — [англійська](./README.md).

> **Самостійно розгорнутий проксі-сервер колегіального міркування для Ollama Cloud, сумісний з OpenAI API та Anthropic Messages API.**  
> Одне віртуальне ім'я моделі запускає панель моделей, суддю і синтезатора — або розумний маршрутизатор, який для кожного запиту вирішує, чи взагалі варта справа такої важкої обробки.

Якщо ви знаєте **OpenRouter Fusion**, ідею ви вже вловили: *думають багато моделей — відповідає одна*.  
**llm-fusion — той самий підхід, перебудований для розробників, яким потрібні контроль, прозорість і безпека агентних циклів.** Він працює локально, маршрутизує запити на Ollama Cloud і додає стратегію `smart`, яку OpenRouter лишає на совісті клієнта.

Без бази даних. Без кроку збірки. Node 24 + TypeScript + Hono, `.ts` виконується напряму через `tsx`. Одна конфігурація YAML.

## Швидкий старт

```bash
git clone https://github.com/Lexus2016/LLM-Fusion.git llm-fusion
cd llm-fusion
npm install                                        # Node ≥ 24
printf 'OLLAMA_API_KEY=ollama-your-key\n' > .env    # ваш ключ Ollama Cloud
npm start                                          # проксі на http://127.0.0.1:8080
```

Далі скеруйте будь-який OpenAI-сумісний клієнт на `http://127.0.0.1:8080/v1` і запитуйте модель за ім'ям — або запустіть OpenCode однією командою:

```bash
./bin/fusion-opencode fusion-coder                # підніме проксі за потреби, під'єднає OpenCode, відкриє TUI
```

Вперше тут? Спершу прочитайте **[Яку модель обрати?](#яку-модель-обрати)**.

## Чому llm-fusion, а не OpenRouter Fusion?

Обидва проганяють запит через панель моделей плюс крок судді/синтезатора, щоб отримати сильнішу відповідь. Різниця — у тому, **де, коли і як** цей конвеєр працює.

| | **llm-fusion** | **OpenRouter Fusion** |
|---|---|---|
| **Розгортання і контроль** | Працює на вашій машині / у Docker. Один користувач, повна прозорість, один файл конфігурації. | Повністю керований SaaS на OpenRouter. |
| **Рахунок** | Платите Ollama Cloud напряму. | Платите OpenRouter; використання агреговане. |
| **Охоплення провайдерів** | Лише Ollama Cloud (один рахунок, один upstream). | Будь-який провайдер на OpenRouter (ширший каталог моделей). |
| **Автоматична маршрутизація** | Вбудована стратегія `smart`: швидкий LLM-маршрутизатор для **кожного запиту** обирає `single` (дешево) чи `fusion` (глибоко). | Fusion завжди запускає повну панель; маршрутизація — вручну або окремим маршрутизатором. |
| **Безпека агентних циклів** | Видає **рівно один `tool_calls`** на крок; панель у режимі `deliberate` ніколи не торкається справжніх інструментів. | Плагін повертає аналіз; фінальний виклик інструмента вирішує модель, що викликає. |
| **Важіль вартості для довгих сесій** | `fusion_planning_turn_only`: повна панель на ході планування, далі лише синтезатор (5 викликів → 1) для механічних кроків усередині циклу. | Немає автоматичного здешевлення окремих запитів усередині циклу. |
| **Веб-заземлення** | Опційне, за явним увімкненням: один пошук Tavily перед панеллю, результати вставляються як текстовий контекст. Умови — `TAVILY_API_KEY` (середовище) **і** `web_search.enabled` (конфігурація); без обох повністю ВИМКНЕНО. | Вбудовані `web_search` + `web_fetch` для панелі та судді. |
| **Стійкість** | Ланцюжки `failover` + плавна деградація + ліміт `max_concurrency` з пер-модельними бюджетами. | Керована надійність + захист від рекурсії. |
| **Стиль конфігурації** | Один YAML з гарячим перезавантаженням та емпірично підібраними пресетами. | Plugin JSON / серверний інструмент / слаг моделі; пресети + веб-інтерфейс. |

**Обирайте llm-fusion, коли:**

- Ваш автономний агент працює **сотні кроків** і потребує детермінованих побічних ефектів рівно один раз.
- Ви хочете, щоб рутинні кроки коштували **1 виклик до провайдера**, а fusion лишався для складних кроків і відновлення після помилок.
- Вам потрібен **локальний, прозорий процес** з одним файлом конфігурації і без додаткової прив'язки до постачальника.
- Ваші моделі та бюджет живуть на **Ollama Cloud**.

**Обирайте OpenRouter Fusion, коли:**

- Потрібні **дослідження із веб-заземленням** на передових моделях багатьох провайдерів.
- Хочеться **керованого сервісу** з веб-майданчиком і глобальною інфраструктурою.
- Ви робите одноразове глибоке дослідження, а не довгі агентні цикли.

**Суть:** OpenRouter Fusion оптимізує стелю якості одиничної відповіді в дослідницьких задачах. **llm-fusion оптимізований для роботи всередині агентного циклу** — контролює вартість, не допускає подвійних викликів інструментів і вмикає важке міркування лише там, де воно справді окупається.

## Яку модель обрати?

У `fusion.yaml` постачаються три **спеціалізовані під задачі** пресети, зібрані за результатами емпіричного відбору моделей (8 моделей × 3 проби — програмування, дослідження, виклик інструментів):

| Викликайте цю модель | Для чого | Стратегія | Як зібрана |
|---|---|---|---|
| **`fusion-coder`** | програмування, планування, аудит коду | `fusion` | панель `glm-5.2` + `kimi-k2.7-code` + `gpt-oss:120b` → суддя `glm-5.2` → синтезатор `glm-5.2` |
| **`fusion-researcher`** | дослідження, аналіз, звіти | `fusion` | панель `kimi-k2.7-code` + `glm-5.2` + `gpt-oss:120b` → суддя `glm-5.2` → синтезатор `kimi-k2.7-code` |
| **`fusion-agents`** | автономні агентні цикли | `smart` | маршрутизатор `glm-5.2`; прості кроки → `glm-5.2`, складні кроки та відновлення після помилок → панель `fusion-coder` |

Два правила випливли прямо з даних:

- **Для програмування — fusion, а не одна модель.** Архітектура і планування справді виграють від кількох точок зору. (Чистий *аудит* коду — просте перелічення проблем — єдина «програмістська» задача, де перемагає одна модель, і вона не репрезентативна для програмування загалом.)
- **Панелі навмисно змішують лінії моделей.** `glm`/`kimi`/`deepseek`/`minimax`/`qwen` — усе китайські лабораторії зі спільними сліпими зонами; кожна панель додає західний декорелятор (`gpt-oss:120b` — лінія OpenAI), щоб помилки панелі корелювали менше — у цьому й сенс панелі. (Початковим декорелятором був Gemini, але він відхиляє історію викликів інструментів, створену іншими моделями, з помилкою `thought_signature` на кожному кроці всередині циклу, тож його замінили.)

Початкові універсальні пресети (`fusion-1`, `smart-1`, `fast-glm` / `fast-kimi` / `fast-deepseek`) також лишаються — для разових потреб.

## Архітектура

Запит до `POST /v1/chat/completions` несе **віртуальне ім'я моделі**. Проксі зіставляє це ім'я з однією з чотирьох стратегій і виконує:

| Стратегія | Що робить |
|----------|--------------|
| `single` | Пряма передача 1:1 до однієї моделі провайдера (потік і без потоку). Примітив, на якому побудовано решту. |
| `failover` | Упорядкований ланцюжок; при збої до першого токена переходить до наступного учасника. Стійкість без розгалуження. |
| `fusion` | Конвеєр міркування: **панель → суддя → синтезатор** (нижче). |
| `smart` | LLM-**маршрутизатор** класифікує кожен запит і скеровує в `single` (дешево) або `fusion` (глибоко). |

### Fusion: панель → суддя → синтезатор

1. **Панель** — запит іде до N моделей *паралельно* (типово 3). У режимі інструментів `deliberate` учасники панелі отримують *описи* інструментів як текст, але не справжню схему `tools`, тож нижче за течією завжди з'являється лише один канонічний виклик інструмента. Опційно один учасник працює **змагальним рецензентом** (мандат «червоної команди»: знайди ваду, обстоюй протилежну позицію) — див. «Відомі обмеження». Довгі контексти стискаються до розгалуження панелі, щоб не переповнювати контекстні вікна учасників.
2. **Суддя** — один виклик зі структурованим JSON порівнює відповіді панелі (консенсус, розбіжності, унікальні спостереження, сліпі зони, часткове покриття, ознаки галюцинацій) і видає **калібровану впевненість** (`high`/`medium`/`low`) разом із `fragile_claims`; синтезатор пом'якшує хиткі та маловпевнені твердження замість того, щоб видавати спільні апріорні уявлення за тверду певність. Якщо суддя повертає некоректний JSON, етап плавно деградує до сирих відповідей панелі, а не завершується помилкою.
3. **Синтезатор** — фінальна модель пише відповідь, яку отримує клієнт, стрімить за запитом і **єдина** отримує справжню схему `tools` (тож до агента доходить рівно один `tool_calls`). Опційно після нього — оцінка фактичної узгодженості **BinEval**, що повертається в заголовках.

### Smart: маршрутизатор

`smart` робить **один** непотоковий виклик із `temperature: 0` і лише-JSON відповіддю до швидкої моделі-`router`, читає `{ "route": "simple" | "fusion" }` і скеровує у відповідну підстратегію. Будь-яка помилка маршрутизатора, таймаут чи нерозбірлива відповідь ведуть до налаштованого маршруту `default` (типово `simple`, бо контроль вартості — головна мета). Підстратегії можна задавати вбудованими блоками або рядковими посиланнями на інші налаштовані моделі. Рішення маршрутизатора кешуються за ідентичним тілом запиту, а одночасні ідентичні запити об'єднуються, тож сплеск однакових ходів повторно використовує один виклик маршрутизатора. Якщо останній результат інструмента схожий на збій (помилка / виняток / ненульовий код виходу), запит іде одразу в `fusion` повз маршрутизатор — відновлення і є тим місцем, де міркування окупається.

### Опційні можливості fusion

Усі вони **типово ВИМКНЕНІ**; вмикаються окремо для кожної fusion-моделі у `fusion.yaml`. Повний довідник: [`fusion.example.yaml`](./fusion.example.yaml).

- **Калібрована впевненість судді** — суддя видає `confidence` (`high`/`medium`/`low`) і `fragile_claims` (спірні / підтверджені одним джерелом / слабкі твердження). Синтезатор пом'якшує такі твердження. Судді **приписано**, що будь-які `hallucination_flags` чи `fragile_claims` мають знижувати `confidence` до `medium`/`low` (це правило на рівні підказки, а не жорстка перевірка в коді — схема приймає JSON судді як є).
- **Змагальний учасник панелі** — `adversarial: <учасник>` перетворює одне місце на рецензента «червоної команди»: обстоювати протилежну позицію, шукати вади, приховані припущення, крайові випадки і стани гонитви замість згоди з консенсусом. Рольова декореляція поверх декореляції за лініями моделей. Змагального учасника **ніколи не скасовують достроково**, панель на нього чекає, тож його відповідь не губиться; йому приписано казати «справжньої проблеми не бачу», коли консенсус міцний. Перевіряється, що це наявний учасник панелі.
- **Веб-заземлення (Tavily)** — `web_search: { enabled: true, max_results, timeout_s, max_context_chars, max_prompt_chars }`. Проксі виконує **один** пошук Tavily до розгалуження панелі і вставляє очищені результати текстовим контекстом кожному учаснику (ніхто не отримує справжні інструменти, тож інваріант одного `tool_calls` зберігається). Умови — `TAVILY_API_KEY` у середовищі *і* `web_search.enabled` на моделі (нема ключа чи увімкнення → нема пошуку, затримки й витрат), **плюс два пропуски під час виконання**: якщо в останньому повідомленні користувача немає придатного тексту запиту або підказка вже перевищує `max_prompt_chars` (типово 80 000) — розмірний запобіжник не дає веб-контексту переповнити учасника з меншим вікном (наприклад, gpt-oss:120b зі 128k) у довгих продовженнях агентного циклу. Контекст вставляється ходом `user` (не `system`) із датою на початку, щоб моделі із застарілою межею знань (kimi-k2.7-code) використовували свіжі факти, а не відмовлялись. Невдалий пошук мовчки деградує до панелі без заземлення. Шанує `fusion_planning_turn_only`. Ключ додайте в `.env`:
  ```bash
  echo 'TAVILY_API_KEY=tvly-...' >> .env
  ```
- **Пост-синтезаторна перевірка BinEval** — `bineval: { enabled: true, model: <eval>, threshold: 0.7, dimensions: [...] }`. Після успішного *непотокового* синтезатора один додатковий виклик оцінювача виставляє бали за фактичну узгодженість (або за вашими бінарними питаннями) і повертає результат у заголовках відповіді: `X-Fusion-Bineval-Score` (0–1), `X-Fusion-Bineval-Dimensions` (JSON за вимірами) і `X-Fusion-Bineval-Low-Score: true`, коли загальний бал нижчий за `threshold`. Якщо BinEval налаштовано, але оцінювання не відбулося, проксі ставить `X-Fusion-Bineval-Skipped: <причина>`, щоб клієнт відрізняв «бал високий» від «оцінка не запускалась» — причини: `streaming`, `synth_error` (синтезатор ≥400), `eval_failed` (оцінювач упав/перевищив час/розімкнене коло), `empty_output` (відповідь лише з інструментами), `non_json_body`, `synth_only` (шлях лише-синтезатор всередині циклу / обхід). **BinEval лише звітує** — він не запускає повторну маршрутизацію чи повторне міркування.

### Надійність агентних циклів (v0.1.16+)

Довгі агентні сесії раніше гинули на чотирьох залежних від навантаження режимах збою; тепер усі чотири закриті структурно:

- **Захист повноти синтезатора з переходом на суддю.** «Мисляча» модель-синтезатор може оголосити себе завершеною, ще перебуваючи посеред плану (порожня відповідь або один вбудований блок `<think>` без артефакту). Захист виявляє це і на потоковому, і на непотоковому шляху, повторює того самого синтезатора один раз із суворою підказкою завершення, а якщо і повтор непридатний (або сам був обрізаний лімітом токенів посеред виклику інструмента) — робить ОДНУ спробу моделлю-суддею, тож жодна окрема модель не може зупинити агентний цикл. Під час тихого відновлення потік несе SSE-коментарі `: keepalive` (кожні 5 с, налаштовується через `FUSION_SYNTH_RECOVERY_PING_MS`), щоб клієнти й проміжні вузли не розривали з'єднання за таймаутом.
- **Чесний `stop_reason` при обрізанні лімітом токенів.** Виклик інструмента, обрізаний `max_tokens`, повідомляється Anthropic-клієнтам як `stop_reason: "max_tokens"` — а не як виконуваний `tool_use` з битим JSON на вході. Якщо ж ліміт спрацював рівно *після* завершення JSON виклику, виклик лишається виконуваним (`tool_use`) — готовий багатохвилинний Write ніколи не викидається.
- **Пер-модельні бюджети конкурентності.** `upstream.per_model_concurrency` дає кожній реальній моделі провайдера власну браму перед глобальним лімітом — сплеск фонових викликів дрібної моделі (Claude Code видає їх по 80–130 за хвилину) стає в чергу біля власної брами замість блокування інтерактивних ходів у голові спільної черги.
- **Розділені класи трафіку.** Лончер `fusion-claude` типово скеровує фонову модель Claude Code (`ANTHROPIC_SMALL_FAST_MODEL`) на `fast-deepseek` — модель, від якої не залежить жодна панель, суддя, синтезатор, маршрутизатор чи simple-маршрут, — тож фонові сплески не можуть вичерпати ліміт моделі, яка пише ваші файли.

## Чесна нотатка про вартість (прочитайте)

Повний fusion виконується на **кожному** кроці. Агентний цикл (читання → міркування → правка → тести → повторне читання…) множить **виклики API моделей** до провайдера:

```
викликів_на_крок = N_панелі + 1 (суддя) + 1 (синтезатор) = 3 + 1 + 1 = 5
```

Типова програмістська задача на **15–25 кроків** отже видає приблизно **75–125 викликів моделей**, якщо fusion увімкнено завжди.

- **Виконання інструментів НЕ множиться** — інструменти запускаються раз на крок (канонічний `tool_calls` видає лише синтезатор). Дії вашого агента не зачеплені.
- **Виклики API моделей МНОЖАТЬСЯ** — на множник «панель + суддя + синтезатор». Це і є ціна, прийнята для v1.

**Пом'якшення (вже в коробці, не в планах): стратегія `smart`.** Скеруйте агента на `fusion-agents` — і рутинні кроки (`read_file`, `grep`, …) підуть дешевим шляхом `single` (1 виклик), а справді складні отримають повний fusion. Інші вбудовані важелі:

- маленька панель за замовчуванням (3),
- глобальний ліміт `max_concurrency` плюс **пер-модельні бюджети конкурентності** (`per_model_concurrency`) — сплеск на одній моделі стає в чергу біля власної брами, не блокуючи інтерактивні ходи,
- стислі таймаути панелі/судді (90 с / 120 с у пресетах, обидва під стелею провайдера ~182 с),
- перемикач `fusion_planning_turn_only` (повна панель на кожному ході планування — тобто на кожному запиті, де останнє повідомлення є свіжою інструкцією користувача, — і деградація до лише-синтезатора — 5 викликів → 1 — тільки на продовженнях із результатом інструмента всередині циклу; нова задача глибоко в довгій сесії все одно отримає панель).

Саме так llm-fusion тримає довгі агентні цикли доступними за ціною, не жертвуючи міркуванням там, де воно важливе.

## Налаштування

Типово проксі завантажує `./fusion.yaml` (шлях перевизначається через `FUSION_CONFIG`). Повна робоча конфігурація:

```yaml
upstream:
  base_url: https://ollama.com
  api_key_env: OLLAMA_API_KEY      # ІМ'Я змінної середовища з ключем — ніколи не сам ключ
server:
  bind: 127.0.0.1
  port: 8080
models:
  fast-glm:                        # single — пряма передача 1:1
    strategy: single
    target: glm-5.2
  fusion-1:                        # fusion — панель -> суддя -> синтезатор
    strategy: fusion
    panel: [glm-5.2, kimi-k2.7-code, deepseek-v4-pro]
    judge: glm-5.2
    synth: deepseek-v4-pro
  smart-1:                         # smart — маршрутизатор обирає single чи fusion
    strategy: smart
    router: glm-5.2
    default: simple
    simple: fast-glm               # рядкове посилання на single-модель вище
    fusion: fusion-1               # рядкове посилання на fusion-модель вище
```

Це базові примітиви. **Постачаний `fusion.yaml`** також визначає три спеціалізовані пресети з розділу [Яку модель обрати?](#яку-модель-обрати) — `fusion-coder`, `fusion-researcher`, `fusion-agents`. Усе незадане отримує задокументовані значення за замовчуванням. **Повністю анотований довідник** — кожен ключ, кожен дефолт, `failover`, `tool_mode`, `fusion_planning_turn_only`, `overrides`, вбудовані smart-блоки — див. [`fusion.example.yaml`](./fusion.example.yaml).

Шпаргалка стратегій для мапи `models:`:

- **`single`** → `{ strategy: single, target: <модель> }`
- **`failover`** → `{ strategy: failover, chain: [<m1>, <m2>, …] }`
- **`fusion`** → `{ strategy: fusion, panel: [...], judge: <модель>, synth: <модель> }`
- **`smart`** → `{ strategy: smart, router: <модель>, default: simple|fusion, simple: <блок-або-посилання>, fusion: <блок-або-посилання> }`

Конфігурація **перезавантажується гаряче**: редагуйте `fusion.yaml` — зміни маршрутизації і моделей застосовуються наживо (некоректна правка відхиляється, попередня конфігурація зберігається). Зміни блока `upstream` — базова адреса, змінна ключа, конкурентність — потребують перезапуску.

## Змінні середовища

Проксі читає звичайні змінні середовища. Також він **автоматично завантажує локальний `.env`** на старті (нативно в Node 24; відсутність файлу не є помилкою).

| Змінна | Обов'язкова | Типово | Призначення |
|----------|----------|---------|---------|
| `OLLAMA_API_KEY` | Так, для живої роботи | — | Bearer-ключ Ollama Cloud. Ім'я — те, на що вказує `upstream.api_key_env` (типово `OLLAMA_API_KEY`). Тримається лише на сервері; клієнтам не надсилається і не логується. |
| `FUSION_PROXY_TOKEN` | Ні | не задано | Якщо в конфігурації є `server.auth_token_env: FUSION_PROXY_TOKEN`, клієнти мусять надсилати `Authorization: Bearer <це значення>`. Не задано ⇒ проксі без автентифікації (localhost, один користувач) і попереджає на старті. |
| `FUSION_CONFIG` | Ні | `./fusion.yaml` | Шлях до файлу конфігурації. |
| `LOG_PRETTY` | Ні | не задано | `LOG_PRETTY=1` вмикає людиночитні логи (інакше — структурований JSON). |
| `LOG_LEVEL` | Ні | `info` | Рівень логів pino (`debug`, `info`, `warn`, …). |
| `FUSION_BIND` | Ні | `server.bind` | Перевизначає адресу прослуховування без редагування конфігурації (використовується Docker-образом). |
| `FUSION_SYNTH_RECOVERY_PING_MS` | Ні | `5000` | Інтервал SSE-коментарів `: keepalive`, що йдуть у потік, поки всередині нього триває повтор відновлення синтезатора. Додатне число; будь-що інше повертає значення за замовчуванням. |

`.env.example` тут закомітити не можна (пісочниця блокує). Створіть власний `.env` у корені проєкту з таким вмістом:

```dotenv
# Обов'язково для будь-якої живої (нетестової) роботи:
OLLAMA_API_KEY=ollama-your-key-here

# Опційно — розкоментуйте, щоб вимагати клієнтський токен (також задайте
# server.auth_token_env: FUSION_PROXY_TOKEN у fusion.yaml):
# FUSION_PROXY_TOKEN=choose-a-long-shared-secret

# Опційні перевизначення:
# FUSION_CONFIG=./fusion.yaml
# LOG_PRETTY=1
```

## Інтеграція з OpenCode

Скеруйте OpenCode на проксі блоком провайдера в `opencode.json`:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "fusion": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Fusion Proxy",
      "options": { "baseURL": "http://127.0.0.1:8080/v1" },
      "models": {
        "fusion-coder": { "name": "Fusion Coder" },
        "fusion-researcher": { "name": "Fusion Researcher" },
        "fusion-agents": { "name": "Fusion Agents" },
        "fusion-1": { "name": "Fusion 1" },
        "smart-1": { "name": "Smart 1" }
      }
    }
  }
}
```

Ідентифікатори моделей мають збігатися з віртуальними іменами у вашому `fusion.yaml`. Якщо задано `FUSION_PROXY_TOKEN` (клієнтська автентифікація), додайте токен для провайдера `fusion` через `opencode auth login` (або ваш `auth.json`), щоб OpenCode надсилав його Bearer-токеном.

Для автономного агентного циклу правильний вибір за замовчуванням — **`fusion-agents`**: рутинні кроки лишаються дешевими (маршрутизатор → швидка одиночна модель), а панель fusion вмикається на складних кроках і відновленні після помилок. Для агента-програміста — **`fusion-coder`**; для досліджень — **`fusion-researcher`**.

### Запуск OpenCode із fusion-моделлю

Із запущеним проксі (`npm start`) і блоком провайдера вище у вашому `opencode.json`
(локальному для проєкту або глобальному `~/.config/opencode/opencode.json`) модель обирається через `-m fusion/<ім'я>`:

```bash
opencode -m fusion/fusion-coder                         # інтерактивний TUI, програмування / планування
opencode -m fusion/fusion-agents                        # інтерактивний TUI, автономна маршрутизація
opencode run -m fusion/fusion-researcher "Summarize X"  # headless, один запит
opencode run -m fusion/fusion-agents "Fix the bug in Y" # headless
```

### Лончер однією командою (`fusion-opencode`)

Сам `ollama` розширити не можна, але репозиторій постачає власний лончер, щоб вам не доводилося
вручну піднімати проксі чи редагувати конфігурації. Він (1) запускає проксі, якщо той ще не працює,
(2) прописує провайдера `fusion` у вашу глобальну конфігурацію OpenCode — ідемпотентно, торкається
лише ключа `fusion` — і (3) запускає OpenCode з обраною моделлю:

```bash
./bin/fusion-opencode fusion-coder             # TUI для програмування / планування
./bin/fusion-opencode fusion-researcher        # TUI для досліджень / звітів
./bin/fusion-opencode fusion-agents            # TUI для автономних агентних циклів
./bin/fusion-opencode fusion-agents run "Fix the failing test in src/foo.ts"   # headless одноразово
```

Додайте його в `PATH`, щоб викликати `fusion-opencode <модель>` з будь-якої теки:

```bash
npm link            # або: ln -s "$PWD/bin/fusion-opencode" ~/.local/bin/fusion-opencode
fusion-opencode fusion-coder
```

Шанує `FUSION_PROXY_URL` (типово `http://127.0.0.1:8080`) і, якщо проксі вимагає клієнтську
автентифікацію, `FUSION_PROXY_TOKEN` (використовується як apiKey, який надсилає OpenCode).

### Інтеграція з Claude Code

Claude Code розмовляє **Anthropic Messages API**. llm-fusion відкриває `POST /v1/messages` на тій
самій базовій адресі, тож Claude Code можна скерувати на корінь проксі кількома змінними середовища.

#### Лончер однією командою (`fusion-claude`)

`bin/fusion-claude` запускає проксі (за потреби), експортує змінні середовища Anthropic і відкриває
Claude Code з обраною моделлю:

```bash
./bin/fusion-claude fusion-agents            # автономні агентні цикли (за замовчуванням)
./bin/fusion-claude fusion-coder             # програмування / планування
./bin/fusion-claude fusion-researcher        # дослідження / звіти
./bin/fusion-claude fusion-agents run "Fix the failing test in src/foo.ts"
```

Він шанує `FUSION_PROXY_URL` і `FUSION_PROXY_TOKEN` так само, як `fusion-opencode`. Ключова
відмінність від OpenAI-сумісних клієнтів — базова адреса: Claude Code використовує **корінь** проксі
(`http://127.0.0.1:8080`), а не `/v1`, бо викликає `/v1/messages`.

Повний посібник із налаштування, довідник змінних середовища та розв'язання проблем:
[`docs/claude-code.md`](./docs/claude-code.md).

### Для AI-агентів (будь-який OpenAI-сумісний клієнт)

Проксі — це готова заміна ендпоінта OpenAI Chat Completions, тож працює будь-який агентний фреймворк, що розмовляє OpenAI, — не лише OpenCode (Continue, Cline, Aider з базовою адресою OpenAI, ваш власний цикл). Під'єднання:

- **Базова адреса:** `http://127.0.0.1:8080/v1`
- **Ключ API:** будь-який непорожній рядок, поки проксі без автентифікації (наприклад, `local-no-auth`); справжнє значення `FUSION_PROXY_TOKEN`, якщо ви ввімкнули клієнтську автентифікацію.
- **Модель:** за задачею — `fusion-coder`, `fusion-researcher` або `fusion-agents` (див. [Яку модель обрати?](#яку-модель-обрати)).

Проксі видає **рівно один** `tool_calls` на крок незалежно від стратегії, тож наявний цикл інструментів агента працює без змін — панель і суддя fusion відбуваються повністю на сервері й невидимі для агента. Для довгих автономних сесій беріть `fusion-agents`: це `smart`, тож повна панель витрачається лише там, де окупається, інакше — швидка одиночна модель, а при помилковому результаті інструмента міркування вмикається автоматично (`escalate_on_tool_error`). Мінімальний прямий виклик:

```bash
curl http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-agents","messages":[{"role":"user","content":"List the open TODOs in this repo"}]}'
```

## Ендпоінти

| Метод і шлях | Призначення |
|---------------|---------|
| `POST /v1/chat/completions` | Головна точка виведення. OpenAI-сумісна; підтримує `stream`, `tools`, `tool_choice` і блоки зображень. Маршрутизується віртуальним іменем `model`. |
| `POST /v1/messages` | Точка Anthropic Messages API для Claude Code. Перекладає блоки вмісту Anthropic у внутрішній OpenAI-конвеєр і назад; підтримує стрімінг та інструменти. |
| `GET /v1/models` | Перелік налаштованих віртуальних моделей (формат списку OpenAI). Додає `context_window` / `supports_vision`, де відомі можливості. |
| `GET /health` | Живучість. `200`, якщо процес працює. Без перевірки провайдера. |
| `GET /ready` | Готовність. `200` лише коли провайдер досяжний і репрезентативна модель виявляється; інакше `503`. |

## Фаза 0 — жива верифікація

Коректність адаптера провайдера тримається на припущеннях щодо Ollama Cloud (Bearer + SSE, виклик інструментів по моделях, виявлення через `/api/show`, формат зображень). Перевірте їх на живому API своїм ключем:

```bash
OLLAMA_API_KEY=ollama-... npm run smoke
```

Це виконує `test/live.smoke.test.ts` проти `https://ollama.com`. Кожна перевірка — окремий тест із чітким результатом:

- **A-1 — Bearer + завершення** *(ПЕРЕВІРЕНО 2026-06-26)*: непотоковий чат до `glm-5.2` повертає відповідь (підтверджує ключ і openai-сумісний шлях).
- **SSE — стрімінг** *(ПЕРЕВІРЕНО 2026-06-26)*: той самий запит зі `stream: true` повертає SSE-фрагменти `data:`.
- **A-2 — інструменти по моделях** *(ПЕРЕВІРЕНО 2026-06-26 — `tool_calls` на всіх: `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`)*: запит із `tools` до кожної з трьох моделей; логує **PASS/FAIL** по моделі (набір не падає, якщо модель без інструментів, — але синтезатором має бути модель зі статусом PASS, бо весь deliberate-конвеєр залежить від її `tool_calls`).
- **A-3 — `/api/show`** *(ПЕРЕВІРЕНО 2026-06-26 — `capabilities[]` + `*.context_length` для всіх трьох)*: підтверджує, що для цих моделей повертаються `capabilities[]` і `*.context_length`, і логує знайдене.
- **A-4/A-5 — зображення** *(ЩЕ ВІДКРИТО — переперевірити на `kimi-k2.7-code`; попередню тестову модель `qwen3-vl:235b` вилучено 2026-06-16)*: надсилає PNG 1×1 через `image_url` моделі із зором (`kimi-k2.7-code` типово; перевизначення — `FUSION_VISION_MODEL`) і логує, чи прийнято формат зображень OpenAI.

Наприкінці — підсумок, за яким ви фіксуєте два налаштування у `fusion.yaml`: `upstream.api_mode` (`openai`, якщо перевірка зображень прийняла формат OpenAI, інакше `native`) і яку модель безпечно ставити fusion-синтезатором (ту, чий рядок A-2 — PASS).

Без `OLLAMA_API_KEY` набір **пропускається**, тож він ніколи не виконується в CI чи у типовому прогоні тестів.

Модульно-інтеграційний набір використовує мок провайдера (перехоплений `fetch`); він покриває валідацію конфігурації, маршрутизацію, розбір можливостей, кожну стратегію, шлюз інструментів, шлюз зображень і smart-маршрутизацію. Живий smoke-тест — єдиний, що торкається мережі, і лише з ключем.

## Відомі обмеження / ще не під'єднано

- **Нативний NDJSON-стрімінг зображень відкладено.** Зір працює на openai-сумісному шляху; потоковий запит із зображенням, що потрапляє на нативний бекенд `/api/chat` (`api_mode: native` + зображення + `stream: true`), повертає чистий `501`, а не напівпід'єднаний потік. Непотоковий нативний зір працює.
- **Поки немає циклічного розподілу, семантичного кешу і маршрутизації за розміром контексту** — це Фаза 6 / майбутнє. (Рішення smart-маршрутизатора *кешуються* за ідентичним тілом запиту, одночасні ідентичні запити об'єднуються — див. `smart` вище, — але немає семантичного/векторного кешу і маршрутизації за кількістю токенів підказки.)
- **BinEval лише звітує.** Низький `X-Fusion-Bineval-Score` не запускає повторне міркування чи повторний синтез — проксі показує бал (або `X-Fusion-Bineval-Skipped`), а рішення ухвалює клієнт. Автоматичний повтор розглядався і свідомо не під'єднаний (посилення вартості на найдорожчих ходах, хибні спрацювання LLM-судді та дрейф моделі-оцінювача як тихий підсилювач регресій).
- **Лише локальна петля.** Типово слухає `127.0.0.1`, без TLS, один користувач (опційний спільний клієнтський токен, а не автентифікація окремих користувачів). Не розраховано на багатокористувацьке відкриття назовні.
- **Failover + стрімінг**: ланцюжок може перемкнутися лише *до* першого токена; збій провайдера посеред потоку стає помилкою потоку (частково надіслану відповідь не можна тихо перезапустити).
