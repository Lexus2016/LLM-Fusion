# llm-fusion — Fusion Proxy

> [English](./README.md) | [Русский](./README.ru.md) | **Українська**
>
> Переклад відповідає релізу v0.1.27. Канонічна версія — [англійська](./README.md).

> **Три моделі сперечаються над вашим запитом, суддя оцінює їхні відповіді, а фінальна модель пише ту, яку ви побачите.**

llm-fusion — це невеликий проксі, який ви запускаєте на власній машині перед [Ollama Cloud](https://ollama.com). Ваш інструмент для роботи з кодом під'єднується до нього так само, як до будь-якої звичайної моделі — той самий API, той самий стрімінг, ті самі виклики інструментів. Різниця схована за ім'ям моделі: попросіть `fusion-coder` — і отримаєте панель із трьох моделей, що відповідають паралельно, суддю, який порівнює сказане ними, і синтезатора, який пише фінальну відповідь. Відповіді на складні питання виходять помітно сильнішими за те, що будь-яка з цих моделей видає поодинці.

Ганяти панель на кожен запит було б дорого, тому є ще `fusion-agents`: швидкий маршрутизатор дивиться на кожен запит і вирішує, чи заслуговує він повної панелі, чи вистачить одного дешевого виклику. Читання файлу отримує один виклик. Відновлення після завалених тестів — усю панель.

Без бази даних, без кроку збірки, без облікових записів. Node 24, один конфігураційний файл YAML і ваш ключ Ollama Cloud. Проксі розмовляє обома діалектами — OpenAI Chat Completions API та Anthropic Messages API, — тож OpenCode, Claude Code, Continue, Cline, Aider або ваш власний агентний цикл працюють без змін.

Знайомі з **OpenRouter Fusion**? Та сама ідея — *думають багато моделей, відповідає одна*, — але на вашій машині, прозоро і з розрахунком на довгі агентні цикли. Повне порівняння [нижче](#llm-fusion-проти-openrouter-fusion).

## Швидкий старт — 5 хвилин

Потрібні дві речі: **Node.js 24 або новіший** та **ключ API [Ollama Cloud](https://ollama.com)**.

```bash
# 1. Заберіть код
git clone https://github.com/Lexus2016/LLM-Fusion.git llm-fusion
cd llm-fusion

# 2. Встановіть залежності (кроку збірки немає — TypeScript виконується напряму через tsx)
npm install

# 3. Покладіть свій ключ Ollama Cloud у .env
printf 'OLLAMA_API_KEY=ollama-your-key\n' > .env

# 4. Запустіть проксі
npm start
```

Готово — проксі слухає на `http://127.0.0.1:8080`. Перевірте, що працює:

```bash
curl http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-coder","messages":[{"role":"user","content":"Say hello"}]}'
```

### Крок 5 — оберіть модель під задачу

Скеруйте будь-який OpenAI-сумісний клієнт на `http://127.0.0.1:8080/v1` (ключ API — будь-який непорожній рядок, наприклад `local-no-auth`) і попросіть один із трьох готових пресетів:

| Просіть | Коли ви… |
|---|---|
| **`fusion-coder`** | пишете код, плануєте функціональність, перевіряєте зміни |
| **`fusion-researcher`** | досліджуєте тему, аналізуєте дані, пишете звіт |
| **`fusion-agents`** | запускаєте автономний агентний цикл на багато кроків |

### Або пропустіть усе це — одна команда

Скрипти з комплекту піднімають проксі, якщо він не запущений, самі пишуть конфігурацію клієнта і відкривають ваш інструмент:

```bash
./bin/fusion-opencode fusion-coder     # OpenCode TUI, під'єднано і готово
./bin/fusion-claude fusion-agents      # Claude Code, під'єднано і готово
```

## Підключення до вашого інструмента

### OpenCode

Найшвидший шлях — скрипт запуску вище. Щоб під'єднати вручну, додайте блок провайдера в `opencode.json` (локально в проєкті або глобально `~/.config/opencode/opencode.json`):

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "fusion": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Fusion Proxy",
      "options": { "baseURL": "http://127.0.0.1:8080/v1" },
      "models": {
        "fusion-coder": { "name": "Fusion Coder" },
        "fusion-researcher": { "name": "Fusion Researcher" },
        "fusion-agents": { "name": "Fusion Agents" }
      }
    }
  }
}
```

Ідентифікатори моделей мають збігатися з віртуальними іменами у вашому `fusion.yaml`. Якщо ви задали `FUSION_PROXY_TOKEN` (автентифікація клієнтів), додайте токен для провайдера з id `fusion` через `opencode auth login` (або ваш `auth.json`), щоб OpenCode надсилав його як Bearer-токен.

Коли проксі запущений, модель обирається через `-m fusion/<name>`:

```bash
opencode -m fusion/fusion-coder                         # інтерактивний TUI, код / планування
opencode -m fusion/fusion-agents                        # інтерактивний TUI, автономна маршрутизація
opencode run -m fusion/fusion-researcher "Summarize X"  # без інтерфейсу, один запит
```

Скрипт `fusion-opencode` робить усі три кроки — піднімає проксі за потреби, записує провайдера `fusion` у вашу глобальну конфігурацію OpenCode (ідемпотентно: він торкається лише ключа `fusion`) і запускає OpenCode:

```bash
./bin/fusion-opencode fusion-researcher        # TUI для досліджень / звітів
./bin/fusion-opencode fusion-agents run "Fix the failing test in src/foo.ts"   # разовий запуск без інтерфейсу
```

Додайте його в `PATH`, щоб викликати з будь-якої теки:

```bash
npm link            # або: ln -s "$PWD/bin/fusion-opencode" ~/.local/bin/fusion-opencode
fusion-opencode fusion-coder
```

Скрипт враховує `FUSION_PROXY_URL` (типово `http://127.0.0.1:8080`) і, якщо ваш проксі вимагає автентифікацію клієнтів, `FUSION_PROXY_TOKEN` (використовується як apiKey, який надсилає OpenCode).

### Claude Code

Claude Code використовує **Anthropic Messages API**. llm-fusion відкриває `POST /v1/messages` на тій самій базовій адресі, тож `bin/fusion-claude` піднімає проксі (за потреби), експортує змінні середовища Anthropic і запускає Claude Code з обраною моделлю:

```bash
./bin/fusion-claude fusion-agents            # автономні агентні цикли (типово)
./bin/fusion-claude fusion-coder             # програмування / планування
./bin/fusion-claude fusion-researcher        # дослідження / звіти
./bin/fusion-claude fusion-agents run "Fix the failing test in src/foo.ts"
```

Він враховує `FUSION_PROXY_URL` і `FUSION_PROXY_TOKEN` точно так само, як `fusion-opencode`. Одна відмінність від OpenAI-сумісних клієнтів: Claude Code використовує **корінь** проксі (`http://127.0.0.1:8080`), а не `/v1`, бо звертається до `/v1/messages`.

Повний посібник із налаштування, довідник змінних середовища та розв'язання проблем: [`docs/claude-code.md`](./docs/claude-code.md).

### Будь-який інший OpenAI-сумісний клієнт

Проксі — це готова точка входу OpenAI Chat Completions, тож працює все, що говорить мовою OpenAI: Continue, Cline, Aider з базовою адресою OpenAI або ваш власний цикл:

- **Базова адреса:** `http://127.0.0.1:8080/v1`
- **Ключ API:** будь-який непорожній рядок, поки проксі без автентифікації (наприклад `local-no-auth`); реальне значення `FUSION_PROXY_TOKEN`, якщо ви ввімкнули автентифікацію клієнтів.
- **Модель:** `fusion-coder`, `fusion-researcher` або `fusion-agents` — див. [Яку модель обрати?](#яку-модель-обрати)

## Оновлення

`llm-fusion` живе в git і не публікується в npm, тож «оновитися» — це підтягнути останні зміни й освіжити залежності:

```bash
git pull                          # перемотка до найновішого коміту / тега
npm install                       # підхопити зміни залежностей
```

Потім перезапустіть проксі. Ваші `fusion.yaml` і `.env` — ваші: `git pull` ніколи не перезаписує локальні зміни в них (у комплектному `fusion.yaml` лежать пресети моделей, які ви могли редагувати, — тож якщо редагували, або закомітьте/сховайте зміни, або тримайте їх у копії, на яку вказує `FUSION_CONFIG`). Щоб зафіксуватися на конкретному релізі замість гілки `main`:

```bash
git fetch --tags
git checkout v0.1.27              # або будь-який тег зі сторінки Releases
npm install
```

`fusion.example.yaml` — повністю анотований довідник, і він оновлюється між релізами: коли виходить нова версія, порівняйте його зі своєю робочою конфігурацією, щоб побачити нові опції. Журнал змін: [`CHANGELOG.md`](./CHANGELOG.md), релізи: <https://github.com/Lexus2016/LLM-Fusion/releases>.

# Технічний довідник

Далі — деталі: як працює конвеєр, скільки він коштує, кожен ключ конфігурації, кожна адреса. Для старту нічого з цього не потрібно — швидкий старт вище і є все налаштування.

## Яку модель обрати?

У `fusion.yaml` є три **спеціалізовані під задачі** пресети, кожен зібраний за результатами емпіричного відбору моделей (8 моделей × 3 тестові задачі — код, дослідження, виклики інструментів):

| Викликайте цю модель | Для чого | Стратегія | Як зібрана |
|---|---|---|---|
| **`fusion-coder`** | програмування, планування, аудит коду | `fusion` | панель `glm-5.2` + `kimi-k2.7-code` + `mistral-large-3:675b` → суддя `glm-5.2` → синтезатор `glm-5.2` |
| **`fusion-researcher`** | дослідження, аналіз, звіти | `fusion` | панель `kimi-k2.7-code` + `glm-5.2` + `gpt-oss:120b` → суддя `glm-5.2` → синтезатор `kimi-k2.7-code` |
| **`fusion-agents`** | автономні агентні цикли | `smart` | маршрутизатор `glm-5.2`; прості кроки → `glm-5.2`, складні кроки та відновлення після помилок → панель `fusion-coder` |

Два правила виросли прямо з даних:

- **Для коду — fusion, а не одна модель.** Архітектура і планування справді виграють від кількох точок зору. (Чистий *аудит* коду — просто перелік проблем — єдина «кодова» задача, де одна модель перемагає, і вона не репрезентативна для програмування.)
- **Панелі навмисно змішують родоводи моделей.** `glm`/`kimi`/`deepseek`/`minimax`/`qwen` — усе китайські лабораторії зі спільними сліпими зонами; кожна панель додає західний декорелятор — `mistral-large-3:675b` (Mistral) у панелі кодера (з v0.1.26+ він також піднімає заявлене контекстне вікно зі 131K до 262K), `gpt-oss:120b` (родовід OpenAI) у панелі дослідника, — щоб помилки панелі були менш корельованими; у цьому весь сенс панелі. (Початковим декорелятором був Gemini, але він відхиляє історію викликів інструментів від інших моделей з помилкою `thought_signature` на кожному кроці всередині циклу, тож його замінили.)

Початкові універсальні пресети (`fusion-1`, `smart-1`, `fast-glm` / `fast-kimi` / `fast-deepseek`) так само в комплекті для разових потреб.

## Архітектура

Запит на `POST /v1/chat/completions` несе **віртуальне ім'я моделі**. Проксі зіставляє це ім'я з однією з чотирьох стратегій і виконує її:

| Стратегія | Що робить |
|----------|--------------|
| `single` | Пряме передавання 1:1 в одну upstream-модель (стрім і без стріму). Примітив, з якого збудовано все інше. |
| `failover` | Упорядкований ланцюжок; при збої до першого токена перемикається на наступного учасника. Стійкість без розгалуження. |
| `fusion` | Конвеєр обмірковування: **панель → суддя → синтезатор** (нижче). |
| `smart` | LLM-**маршрутизатор** класифікує кожен запит і скеровує його в `single` (дешево) або `fusion` (глибоко). |

### Fusion: панель → суддя → синтезатор

1. **Панель** — запит іде до N моделей *паралельно* (типово 3). У режимі інструментів `deliberate` учасники панелі отримують *описи* інструментів як звичайний текст, але не реальну схему `tools`, тож далі завжди йде лише один канонічний виклик інструмента. За бажанням один учасник працює **змагальним рецензентом** (мандат «червоної команди»: знайди ваду, обстоюй протилежну позицію) — див. «Відомі обмеження». Довгі контексти стискаються перед розгалуженням панелі, щоб не переповнити контекстні вікна учасників.
2. **Суддя** — один виклик зі структурованим JSON порівнює відповіді панелі (консенсус, розбіжності, унікальні спостереження, сліпі зони, часткове покриття, hallucination_flags) і видає **калібровану `confidence`** (`high`/`medium`/`low`) плюс `fragile_claims`; синтезатор пом'якшує крихкі та малодостовірні твердження замість того, щоб видавати спільні упередження за впевненість. Якщо суддя повертає некоректний JSON, етап м'яко деградує до сирих відповідей панелі, а не падає.
3. **Синтезатор** — фінальна модель пише відповідь, яку отримує клієнт, стрімить, коли просять, і **єдина** отримує реальну схему `tools` (тож до агента доходить рівно один `tool_calls`). Опційно після нього йде оцінка фактичної узгодженості **BinEval**, що повертається в заголовках.

### Smart: маршрутизатор

`smart` робить **один** нестрімовий, `temperature: 0`, суто JSON-виклик до швидкої моделі-`router`, читає `{ "route": "simple" | "fusion" }` і скеровує запит у відповідну підстратегію. Будь-яка помилка маршрутизатора, таймаут чи нерозбірлива відповідь відкочуються до налаштованого маршруту `default` (типово `simple`, бо контроль витрат — це і є суть). Підстратегії можуть бути вбудованими блоками або рядковими посиланнями на інші налаштовані моделі, щоб не повторюватися. Рішення маршрутизатора кешуються для ідентичних тіл запитів, а одночасні ідентичні запити об'єднуються, тож сплеск однакових ходів повторно використовує один раунд маршрутизатора. Якщо останній результат інструмента виглядає як збій (error / exception / ненульовий код виходу), запит ескалюється прямо у `fusion`, оминаючи маршрутизатор: відновлення — саме те місце, де обмірковування окупається.

### Необов'язкові функції fusion

Усі вони **типово ВИМКНЕНІ**; вмикаються для кожної fusion-моделі окремо у `fusion.yaml`. Повний довідник: [`fusion.example.yaml`](./fusion.example.yaml).

- **Калібрована впевненість судді** — суддя видає `confidence` (`high`/`medium`/`low`) та `fragile_claims` (спірні / підтверджені лише одним джерелом / слабкі твердження). Синтезатор пом'якшує крихкі та малодостовірні твердження замість видавати спільні упередження з навчання за впевненість. Судді **приписано інструкцією**, що будь-які `hallucination_flags` чи `fragile_claims` мають знижувати `confidence` до `medium`/`low` (це правило на рівні промпту, якого модель-суддя має дотримуватися, а не жорстка перевірка в коді — схема приймає JSON судді як є).
- **Змагальний учасник панелі** — `adversarial: <panel-member>` перетворює одне місце на рецензента «червоної команди»: обстоювати протилежну позицію, шукати вади, приховані припущення, крайні випадки та стани гонитви замість погоджуватися з консенсусом. Рольова декореляція поверх декореляції за родоводом. Змагальний учасник **ніколи не скасовується достроково**, панель на нього чекає, тож його відповідь ніколи не губиться; йому сказано відповідати «I cannot find a real problem», коли консенсус міцний. Перевіряється, що це наявний учасник панелі.
- **Веб-заземлення (Tavily)** — `web_search: { enabled: true, max_results, timeout_s, max_context_chars, max_prompt_chars }`. Проксі виконує **один** пошук Tavily перед розгалуженням панелі і вставляє очищені результати як текстовий контекст кожному учаснику (жоден не отримує реальних інструментів, тож інваріант «один `tool_calls`» зберігається). Умови вмикання: `TAVILY_API_KEY` в середовищі *та* `web_search.enabled` на моделі (немає ключа і немає згоди → немає пошуку, затримки й витрат), **плюс два пропуски під час виконання**: якщо в останньому повідомленні користувача немає придатного тексту запиту або промпт уже перевищує `max_prompt_chars` (типово 80 000) — обмеження за розміром не дає веб-контексту переповнити учасника панелі з меншим вікном (наприклад, gpt-oss:120b зі 128k) у довгих продовженнях агентного циклу. Контекст вставляється як хід `user` (не `system`) і починається з поточної дати, тож моделі із застарілою межею знань (kimi-k2.7-code) користуються свіжими фактами замість відмовлятися. Невдалий пошук тихо деградує до незаземленої панелі. Поважає `fusion_planning_turn_only`. Ключ — у `.env`:
  ```bash
  echo 'TAVILY_API_KEY=tvly-...' >> .env
  ```
- **Пост-синтезова перевірка якості BinEval** — `bineval: { enabled: true, model: <eval>, threshold: 0.7, dimensions: [...] }`. Після успішного *нестрімового* синтезу один додатковий виклик оцінювача виставляє відповіді бал за фактичну узгодженість (або за вашими власними бінарними питаннями) і повертає результати в заголовках відповіді: `X-Fusion-Bineval-Score` (0–1), `X-Fusion-Bineval-Dimensions` (JSON за вимірами) та `X-Fusion-Bineval-Low-Score: true`, коли загальний бал нижчий за `threshold`. Коли bineval налаштований, але оцінювання не відбулося, проксі ставить `X-Fusion-Bineval-Skipped: <причина>`, щоб клієнт міг відрізнити «бал високий» від «оцінювання не було» — причини: `streaming`, `synth_error` (синтез ≥400), `eval_failed` (оцінювач помилився / вийшов за час / розімкнене коло), `empty_output` (відповідь лише з інструментами), `non_json_body`, `synth_only` (шлях planning-turn-only всередині циклу / обхід). **BinEval лише звітує** — він не запускає перемаршрутизацію чи повторне обмірковування.

### Надійність агентних циклів (v0.1.16+)

Довгі агентні прогони раніше гинули на чотирьох залежних від навантаження режимах збою; тепер усі чотири закриті структурно:

- **Захист повноти синтезу із запасним суддею.** «Мисляча» модель-синтезатор може оголосити себе завершеною, ще перебуваючи посеред плану (порожня відповідь або один вбудований блок `<think>` без артефакту). Захист виявляє це і на стрімовому, і на нестрімовому шляхах, повторює той самий синтез один раз зі строгою вимогою завершити і — якщо повтор так само непридатний (або сам обрізаний лімітом токенів посеред виклику інструмента) — робить ОДНУ спробу на іншій моделі (судді, а коли суддя і є синтезатором — учаснику панелі), тож жодна окрема модель не може зупинити агентний цикл. Під час тихого відновлення стрім несе SSE-коментарі `: keepalive` (кожні 5 с, налаштовується через `FUSION_SYNTH_RECOVERY_PING_MS`), щоб клієнти й посередники не відпадали за таймаутом.
- **Чесний `stop_reason` при обрізанні лімітом токенів.** Виклик інструмента, обрізаний `max_tokens`, повідомляється Anthropic-клієнтам як `stop_reason: "max_tokens"` — а не як нібито придатний до виконання `tool_use` з поламаним JSON на вході. Коли ліміт припадає точно *після* завершення JSON виклику, виклик лишається придатним (`tool_use`), тож завершений багатохвилинний Write ніколи не викидається.
- **Бюджети конкурентності на модель.** `upstream.per_model_concurrency` дає кожній реальній upstream-моделі власний шлюз перед глобальним обмежувачем: сплеск фонових викликів маленької моделі (Claude Code видає 80–130/хв) шикується у власну чергу замість блокувати інтерактивні ходи на початку спільної.
- **Розділені класи трафіку.** Скрипт `fusion-claude` типово ставить фонову модель Claude Code (`ANTHROPIC_SMALL_FAST_MODEL`) у `fast-deepseek` — модель, від якої не залежить жодна панель, суддя, синтезатор, маршрутизатор чи простий маршрут, — тож фонові сплески не можуть обмежити швидкість моделі, яка пише ваші файли.
- **Захист інструментальних ходів одиночного маршруту (v0.1.26).** Пряме передавання smart `simple` — маршрут, яким іде ~87 % агентних кроків, — отримало власний захист повноти. Три підтверджені на практиці режими збою раніше завершували хід агента без жодної виконаної дії (той самий стан «зробив один крок і стоїть, поки не наберете *continue*»): модель *переказує* наступну дію прозою замість видати виклик інструмента; запис великого файлу обрізається лімітом виводу посеред аргументів (`finish_reason:"length"`, непридатний JSON); upstream рве довгі потоки генерації (~5 хв на Ollama Cloud). Захист виявляє всі три на запитах з інструментами і виконує один повтор відновлення **з живим стрімом** (з підказкою видати виклик інструмента і писати великі корисні навантаження шматками), а при невдачі віддає початкову відповідь. Кожен стрім з інструментами також пише в лог один рядок `tool-turn terminal state`, тож реальне зависання сесії діагностується з самого лише лога.
- **`request_overrides` на модель (v0.1.26).** Одиночні моделі (та вбудований слот smart `simple`) приймають додаткові поля тіла запиту, що підмішуються в кожен upstream-виклик — наприклад, `request_overrides: { reasoning_effort: "none" }` зупиняє «мислячу» модель від багатохвилинних роздумів на механічних агентних кроках (виміряно A/B на glm-5.2: міркування 1692→0 символів, 6 с→2 с, виклики інструментів неушкоджені; `think:false` і `"low"` Ollama Cloud ігнорує). Ключові поля (`model`, `messages`, `stream`, `tools`, `tool_choice`) захищені. Комплектний пресет `fusion-agents` використовує саме це.

## Конектори, мультиакаунтний failover і панель

Типово проксі працює з одним upstream (один акаунт Ollama Cloud). Натомість
можна задати **впорядкований пул конекторів** — кілька акаунтів Ollama Cloud
та/або інші OpenAI-сумісні провайдери — і проксі **автоматично перемикається
між ними**, коли один деградує, вичерпав ліміт або в нього скінчилася оплата.

```yaml
connectors:
  - id: ollama-primary          # пробується першим
    provider: ollama
    base_url: https://ollama.com
    api_key_env: OLLAMA_API_KEY
  - id: ollama-backup           # ті самі моделі, інший акаунт/ключ
    provider: ollama
    base_url: https://ollama.com
    api_key_env: OLLAMA_API_KEY_2
  - id: openrouter-1            # будь-який OpenAI-сумісний провайдер
    provider: openai-compat
    base_url: https://openrouter.ai/api/v1
    api_key_env: OPENROUTER_API_KEY
    model_map: { "qwen3-coder:480b": "qwen/qwen3-coder" }
```

**Як вирішується failover.** Обслуговує перший здоровий конектор. `429`
(ліміт), `5xx`, мережева помилка чи таймаут **охолоджують** цей конектор на
`connector_cooldown_s`, і естафету перебирає наступний; охолоджений конектор
автоматично перевіряється проб-запитом після вікна очікування. `401` (поганий
ключ), `402` (скінчились кошти) або збіг маркера квоти в тілі позначають його
як **down** («оплата закінчилася») — він пропускається до `connector_down_recheck_s`
або ручного скидання. Коли задроселено кілька конекторів, проксі віддає нагору
найвідновлюванішу помилку (тимчасовий `429` важливіший за `401` мертвого
backup), тож успішний запит ніколи не перетворюється на жорсткий збій. Це
прозоро: кожна модель/стратегія (single, failover, fusion, smart) їде на пулі
без змін конфігу.

**Будь-який OpenAI-сумісний провайдер — лише конфігом.** Узагальнений тип
`openai-compat` покриває OpenRouter, DeepInfra, Together, Novita, Nebius, Groq,
Cerebras, DeepSeek, Mistral, Baseten, … — задайте `base_url`, `api_key_env` та
`model_map` (їхні ID моделей відрізняються від Ollama). Ранжоване порівняння і
те, які провайдери чисто сигналять про вичерпаний акаунт (`402`), — у
[`docs/providers-research.md`](docs/providers-research.md).

**Панель.** Відкрийте **http://127.0.0.1:8080/panel**, щоб бачити все наживо:
який конектор активний, які охолоджуються чи вимкнені та *чому* (причина +
остання помилка + зворотний відлік), лічильники запитів/помилок на конектор, а
також ручне керування — вимкнути, увімкнути, скинути чи закріпити конектор
активним. Коли задано клієнтський auth-токен, дані та дії панелі вимагають його
(HTML-оболонка не містить секретів). Зворотна сумісність: без `connectors:`
працює як раніше — старий одинарний `upstream.base_url` + `api_key_env`.

## Чесна нотатка про вартість (прочитайте це)

Повний fusion працює на **кожному** кроці. Агентний цикл (прочитати → подумати → відредагувати → прогнати тести → перечитати…) множить кількість upstream-**викликів моделей**:

```
upstream_calls_per_step = N_panel + 1 (суддя) + 1 (синтезатор) = 3 + 1 + 1 = 5
```

Типова задача з кодом на **15–25 кроків** отже видає приблизно **75–125 upstream-викликів моделей**, якщо fusion увімкнений завжди.

- **Виконання інструментів НЕ множиться** — інструменти запускаються раз на крок (канонічний `tool_calls` видає лише синтезатор). Дії вашого агента не зачеплені.
- **Виклики API моделей множаться** — на множник панель + суддя + синтезатор. Це і є ціна, і для v1 вона прийнята свідомо.

**Пом'якшення (в комплекті, не «на потім»): стратегія `smart`.** Скеруйте агента на `fusion-agents` — і рутинні кроки (`read_file`, `grep`, …) підуть дешевим шляхом `single` (1 виклик), а справді складні так само отримають повний fusion. Інші вбудовані важелі:

- маленька типова панель (3),
- глобальний ліміт `max_concurrency` плюс **бюджети конкурентності на модель** (`per_model_concurrency`) — сплеск на одній моделі шикується у власну чергу замість блокувати інтерактивні ходи,
- жорсткі таймаути панелі/судді (90 с / 120 с у комплектних пресетах, обидва нижчі за upstream-стелю ~182 с),
- перемикач `fusion_planning_turn_only` (повна панель на кожному плановому ході — будь-якому запиті, де останнє повідомлення є свіжою інструкцією користувача, — і деградація до самого лише синтезу — 5 викликів → 1 — тільки на продовженнях із результатами інструментів усередині циклу; нова задача глибоко в довгій сесії все одно отримає панель).

Саме так llm-fusion тримає довгі агентні цикли доступними за грошима, не жертвуючи обмірковуванням там, де воно важить.

## llm-fusion проти OpenRouter Fusion

Обидва проганяють запит через панель моделей плюс крок судді/синтезатора, щоб отримати сильнішу відповідь. Різниця — у тому, **де, коли і як** цей конвеєр працює.

| | **llm-fusion** | **OpenRouter Fusion** |
|---|---|---|
| **Розміщення і контроль** | Працює на вашій машині / у Docker. Один користувач, можна зазирнути всередину, один файл конфігурації. | Повністю керований SaaS на OpenRouter. |
| **Рахунок** | Платите Ollama Cloud напряму. | Платите OpenRouter; використання йде пакетом. |
| **Охоплення провайдерів** | Лише Ollama Cloud (один рахунок, один upstream). | Будь-який провайдер на OpenRouter (ширший каталог моделей). |
| **Автоматична маршрутизація** | Вбудована стратегія `smart`: швидкий LLM-маршрутизатор обирає `single` (дешево) чи `fusion` (глибоко) **для кожного запиту**. | Fusion завжди жене повну панель; маршрутизація ручна або окремим маршрутизатором. |
| **Безпека агентних циклів** | Видає **рівно один `tool_calls`** на крок; панель ніколи не торкається реальних інструментів у режимі `deliberate`. | Плагін повертає аналіз; фінальне використання інструментів вирішує модель, що викликає. |
| **Важіль вартості для довгих прогонів** | `fusion_planning_turn_only`: повна панель на плановому ході, далі лише синтез (5 викликів → 1) для механічних кроків усередині циклу. | Немає автоматичного пониження окремих запитів усередині циклу. |
| **Веб-заземлення** | Опційне, за явною згодою: один пошук Tavily перед панеллю, результати вставляються текстовим контекстом. Умови: `TAVILY_API_KEY` (середовище) **і** `web_search.enabled` (конфігурація) — повністю ВИМКНЕНО, поки не задані обидва. | Вбудовані `web_search` + `web_fetch` для панелі та судді. |
| **Стійкість** | Ланцюжки `failover` + м'яка деградація + ліміт `max_concurrency`. | Керована надійність + захист від рекурсії. |
| **Стиль конфігурації** | Один YAML з гарячим перезавантаженням та емпірично підібраними пресетами. | JSON плагіна / серверний інструмент / slug моделі; пресети + веб-інтерфейс. |

**Беріть llm-fusion, коли:**

- Ви ганяєте автономного агента **сотні кроків** і вам потрібні детерміновані побічні ефекти рівно-один-раз.
- Ви хочете, щоб рутинні кроки коштували **1 upstream-виклик**, а fusion лишався для складних кроків і відновлення після помилок.
- Вам ближчий **локальний, прозорий процес** з одним файлом конфігурації і без додаткової прив'язки до постачальника.
- Ваші моделі та бюджет живуть на **Ollama Cloud**.

**Беріть OpenRouter Fusion, коли:**

- Потрібні **дослідження з веб-заземленням** на передових моделях багатьох провайдерів.
- Хочете **керований сервіс** з веб-майданчиком і глобальною інфраструктурою.
- Ви робите разове глибоке дослідження, а не довгі агентні цикли.

**Суть:** OpenRouter Fusion оптимізує стелю якості разової відповіді на дослідницьких задачах. **llm-fusion оптимізує роботу всередині агентного циклу** — контроль витрат, запобігання подвійним викликам інструментів і важке обмірковування лише там, де воно справді окупається.

## Конфігурація

Типово проксі завантажує `./fusion.yaml` (шлях можна змінити через `FUSION_CONFIG`). Ось скорочена копія комплектної конфігурації — три спеціалізовані пресети плюс одна single-модель:

```yaml
upstream:
  base_url: https://ollama.com
  api_key_env: OLLAMA_API_KEY        # ІМ'Я змінної середовища з ключем — ніколи не сам ключ
server:
  bind: 127.0.0.1
  port: 8080
models:
  # single — пряме передавання 1:1; викликайте, коли потрібна одна швидка модель без панелі
  fast-glm:
    strategy: single
    target: glm-5.2

  # fusion — панель -> суддя -> синтезатор; програмування, планування, аудит коду
  fusion-coder:
    strategy: fusion
    panel: [glm-5.2, kimi-k2.7-code, mistral-large-3:675b]
    judge: glm-5.2
    synth: glm-5.2
    tool_mode: deliberate

  # fusion — панель під дослідження і звіти, опційне веб-заземлення
  fusion-researcher:
    strategy: fusion
    panel: [kimi-k2.7-code, glm-5.2, gpt-oss:120b]
    judge: glm-5.2
    synth: kimi-k2.7-code
    tool_mode: deliberate
    web_search:
      enabled: true                  # також потрібен TAVILY_API_KEY у .env, інакше повністю ВИМКНЕНО
      max_results: 3

  # smart — маршрутизатор обирає дешевий single чи повний fusion на кожен запит; агентні цикли
  fusion-agents:
    strategy: smart
    router: glm-5.2
    default: simple
    escalate_on_tool_error: true     # збійний результат інструмента йде прямо у fusion
    simple:
      target: glm-5.2                # механічні кроки (read / grep / run)
      request_overrides:
        reasoning_effort: "none"     # без багатохвилинних роздумів на механічних кроках
    fusion: fusion-coder             # складні кроки -> панель кодера (рядкове посилання)
```

Викликається будь-що з цього через поле `model` у запиті — `fast-glm` відповість однією моделлю без панелі. Повний комплектний [`fusion.yaml`](./fusion.yaml) додатково містить `fast-kimi` / `fast-deepseek` / `fusion-1` / `smart-1` для разових потреб, підібрані таймаути, бюджети конкурентності на модель і блоки BinEval. Усе, що не задано, відкочується до задокументованих типових значень. **Повністю анотований довідник** — кожен ключ, кожне типове значення, `failover`, `tool_mode`, `fusion_planning_turn_only`, `overrides`, вбудовані підблоки smart — див. [`fusion.example.yaml`](./fusion.example.yaml).

Шпаргалка стратегій для мапи `models:`:

- **`single`** → `{ strategy: single, target: <model> }`
- **`failover`** → `{ strategy: failover, chain: [<m1>, <m2>, …] }`
- **`fusion`** → `{ strategy: fusion, panel: [...], judge: <model>, synth: <model> }`
- **`smart`** → `{ strategy: smart, router: <model>, default: simple|fusion, simple: <single-блок-або-посилання>, fusion: <fusion-блок-або-посилання> }`

Конфігурація **перезавантажується гаряче**: відредагуйте `fusion.yaml` — і зміни маршрутизації/моделей застосуються наживо (некоректна правка відхиляється, попередня конфігурація зберігається). Зміна блоку `upstream` — базова адреса, змінна ключа, конкурентність — потребує перезапуску.

## Змінні середовища

Проксі читає звичайні змінні середовища. Він також **автоматично підвантажує локальний `.env`** на старті (нативно в Node 24; відсутність файлу — не проблема).

| Змінна | Обов'язкова | Типово | Призначення |
|----------|----------|---------|---------|
| `OLLAMA_API_KEY` | Так, для живої роботи | — | Bearer-ключ Ollama Cloud. Ім'я — те, на що вказує `upstream.api_key_env` (типово `OLLAMA_API_KEY`). Тримається лише на боці сервера; ніколи не надсилається клієнтам і не потрапляє в лог. |
| `FUSION_PROXY_TOKEN` | Ні | не задано | Коли в конфігурації задано `server.auth_token_env: FUSION_PROXY_TOKEN`, клієнти мусять надсилати `Authorization: Bearer <це значення>`. Не задано ⇒ проксі без автентифікації (localhost, один користувач) і попереджає про це на старті. |
| `FUSION_CONFIG` | Ні | `./fusion.yaml` | Шлях до файлу конфігурації. |
| `LOG_PRETTY` | Ні | не задано | `LOG_PRETTY=1` вмикає зручні для читання логи (інакше — структурований JSON). |
| `LOG_LEVEL` | Ні | `info` | Рівень логів pino (`debug`, `info`, `warn`, …). |
| `FUSION_BIND` | Ні | `server.bind` | Перевизначає адресу прослуховування без редагування конфігурації (використовується Docker-образом). |
| `FUSION_SYNTH_RECOVERY_PING_MS` | Ні | `5000` | Інтервал SSE-коментарів `: keepalive`, що йдуть у стрім, поки триває повтор відновлення синтезу. Додатне число; будь-що інше відкочується до типового. |

`.env.example` тут закомітити не можна (обмеження пісочниці). Створіть власний `.env` у корені проєкту з таким буквальним вмістом:

```dotenv
# Обов'язково для будь-якої живої (не тестової) роботи:
OLLAMA_API_KEY=ollama-your-key-here

# Опційно — розкоментуйте, щоб вимагати клієнтський токен (також задайте
# server.auth_token_env: FUSION_PROXY_TOKEN у fusion.yaml):
# FUSION_PROXY_TOKEN=choose-a-long-shared-secret

# Опційні перевизначення:
# FUSION_CONFIG=./fusion.yaml
# LOG_PRETTY=1
```

## Запуск

```bash
npm run dev      # tsx watch — перезапуск при зміні коду
npm run start    # tsx — разовий запуск
```

Або з ключем прямо в команді (без `.env`):

```bash
OLLAMA_API_KEY=ollama-... npm run start
```

На старті друкується банер: адреса прослуховування, завантажені віртуальні моделі з їхніми стратегіями, чи ввімкнена автентифікація клієнтів і чи присутній upstream-ключ.

## Endpoints

| Метод і шлях | Призначення |
|---------------|---------|
| `POST /v1/chat/completions` | Головна точка входу. Сумісна з OpenAI; підтримує `stream`, `tools`, `tool_choice` і блоки зображень. Маршрутизується за віртуальним іменем `model`. |
| `POST /v1/messages` | Точка входу Anthropic Messages API для Claude Code. Перекладає блоки вмісту Anthropic у внутрішній OpenAI-конвеєр і назад; підтримує стрімінг та інструменти. |
| `GET /v1/models` | Перелік налаштованих віртуальних моделей (формат списку OpenAI). Додає `context_window` / `supports_vision` там, де можливості моделі відомі. |
| `GET /health` | Життєздатність. `200`, якщо процес живий. Без перевірки upstream. |
| `GET /ready` | Готовність. `200` лише якщо upstream досяжний і репрезентативна модель знаходиться; інакше `503`. |
| `GET /panel` | Локальна панель конекторів (HTML). Показує, який конектор активний, які охолоджуються/вимкнені та чому, з ручним керуванням. |
| `GET /admin/connectors` | JSON-знімок здоровʼя конекторів (під auth-токеном, якщо його задано). Живить панель. |
| `POST /admin/connectors/:id/{disable,enable,reset,pin}` · `POST /admin/unpin` | Ручне керування конекторами (під auth-токеном). |

## Фаза 0 — жива верифікація

Коректність адаптера upstream тримається на припущеннях про Ollama Cloud (Bearer + SSE, виклики інструментів для кожної моделі, виявлення через `/api/show`, формат зображень). Перевірте їх на живому API своїм ключем:

```bash
OLLAMA_API_KEY=ollama-... npm run smoke
```

Це запускає `test/live.smoke.test.ts` проти `https://ollama.com`. Кожна перевірка — окремий тест із чітким результатом у лозі:

- **A-1 — Bearer + відповідь** *(ПЕРЕВІРЕНО 2026-06-26)*: нестрімовий чат до `glm-5.2` повертає відповідь (підтверджує ключ і openai-сумісний шлях).
- **SSE — стрімінг** *(ПЕРЕВІРЕНО 2026-06-26)*: той самий запит зі `stream: true` повертає SSE-шматки `data:`.
- **A-2 — виклики інструментів по моделях** *(ПЕРЕВІРЕНО 2026-06-26 — `tool_calls` на всіх: `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`)*: запит із `tools` до кожної з `glm-5.2`, `kimi-k2.7-code`, `deepseek-v4-pro`; у лог іде **PASS/FAIL** для кожної (суїта *не* падає, якщо модель без інструментів — але синтезатором має бути модель з PASS, бо весь конвеєр обмірковування залежить від її `tool_calls`).
- **A-3 — `/api/show`** *(ПЕРЕВІРЕНО 2026-06-26 — `capabilities[]` + `*.context_length` повернулися для всіх трьох)*: підтверджує, що для цих моделей приходять `capabilities[]` і `*.context_length`, і логує знайдене.
- **A-4/A-5 — зображення** *(ЩЕ ВІДКРИТО — переперевірити на `kimi-k2.7-code`; попередню тестову модель `qwen3-vl:235b` зняли 2026-06-16)*: надсилає PNG 1×1 як `image_url` моделі із зором (`kimi-k2.7-code` типово; змінюється через `FUSION_VISION_MODEL`) і логує, чи прийнято формат зображень OpenAI.

Суїта друкує підсумок, за яким ви фіксуєте два налаштування у `fusion.yaml`: `upstream.api_mode` (`openai`, якщо перевірка зображень прийняла формат OpenAI, інакше `native`) і яку модель безпечно ставити fusion-`synth` (ту, чий рядок A-2 — PASS).

Без `OLLAMA_API_KEY` суїта **пропускається**, тож вона ніколи не бігає в CI чи типовому прогоні тестів.

## Тестування

```bash
npm test          # vitest run — швидко, офлайн, без ключа й мережі (живий smoke пропускається)
npm run typecheck # tsc --noEmit
```

Модульна/інтеграційна суїта використовує мок-upstream (перехоплений `fetch`); вона покриває валідацію конфігурації, маршрутизацію, розбір можливостей, кожну стратегію, шлюз інструментів, шлюз зображень і smart-маршрутизацію. Живий smoke-тест — єдиний, що торкається мережі, і лише за наявності ключа.

## Відомі обмеження / ще не підключено

- **Нативний NDJSON-стрімінг зображень відкладено.** Зір працює на openai-сумісному шляху; стрімовий запит із зображенням, який потрапляє на нативний бекенд `/api/chat` (`api_mode: native` + зображення + `stream: true`), повертає чистий `501`, а не напівпідключений стрім. Нестрімовий нативний зір працює.
- **Ще немає round-robin, семантичного кешу і маршрутизації за розміром контексту** — це Фаза 6 / майбутнє. (Рішення smart-маршрутизатора *кешуються* для ідентичних тіл запитів, а одночасні ідентичні запити об'єднуються — див. `smart` вище, — але семантичного/векторного кешу і маршрутизації за кількістю токенів немає.)
- **BinEval лише звітує.** Низький `X-Fusion-Bineval-Score` не запускає повторне обмірковування чи повторний синтез — проксі показує бал (або `X-Fusion-Bineval-Skipped`), а рішення за клієнтом. Опційний авто-повтор розглядали і свідомо не підключили (примноження вартості на найдорожчих ходах, хибнонегативні LLM-судді і дрейф моделі-оцінювача як тихий підсилювач регресій).
- **Лише loopback.** Типово слухає `127.0.0.1`, без TLS, один користувач (опційний спільний клієнтський токен, а не автентифікація окремих користувачів). Не для багатокористувацького виставляння назовні.
- **Failover + стрімінг**: ланцюжок може перемкнутися лише *до* першого токена; збій upstream посеред стріму виринає як помилка стріму (частково надіслану відповідь тихо перезапустити не можна).

## Для AI-агентів

Ви — AI-агент, якому треба під'єднати себе (або обгортку, в якій ви працюєте) до llm-fusion. Цей розділ самодостатній: для робочого налаштування більше нічого в цьому файлі читати не потрібно.

### 1. Перевірте, чи проксі вже працює

```bash
curl -sf http://127.0.0.1:8080/health   # 200 → процес живий
curl -sf http://127.0.0.1:8080/ready    # 200 → upstream досяжний; 503 → проблема з upstream
```

Якщо не працює — запустіть із кореня репозиторію:

```bash
cd <шлях-до-llm-fusion> && npm start
```

Для старту потрібен файл `.env` з `OLLAMA_API_KEY=<ключ>`. Якщо ключа немає, попросіть його в людини-оператора — ніколи не вигадуйте ключ і не переносьте його з іншого місця.

### 2. Оберіть точку входу під свій діалект API

- **Діалект OpenAI Chat Completions** → базова адреса `http://127.0.0.1:8080/v1`, запити на `POST /v1/chat/completions`.
- **Діалект Anthropic Messages** (Claude Code і подібні) → базова адреса — **корінь** проксі `http://127.0.0.1:8080` (не `/v1`), запити на `POST /v1/messages`.
- Доступні віртуальні імена моделей дізнавайтеся під час роботи з `GET /v1/models` — це краще за жорстке кодування, бо `fusion.yaml` перезавантажується гаряче і список моделей може змінитися, поки проксі працює.

### 3. Автентифікація

- Типово проксі без автентифікації на localhost. Надсилайте будь-який непорожній рядок як ключ API (наприклад `local-no-auth`) — деякі клієнтські SDK відмовляються працювати з порожнім ключем.
- Якщо оператор увімкнув автентифікацію клієнтів (`FUSION_PROXY_TOKEN` + `server.auth_token_env` у `fusion.yaml`): надсилайте `Authorization: Bearer <токен>`. Відповідь `401` означає, що токен потрібен або хибний.

### 4. Оберіть модель під задачу

| Модель | Для чого |
|---|---|
| `fusion-agents` | **Типова для автономних циклів.** Розумна маршрутизація: рутинні кроки коштують 1 upstream-виклик; складні кроки і відновлення після помилок автоматично отримують повну панель. |
| `fusion-coder` | Ходи з кодом і плануванням, де панель потрібна завжди. |
| `fusion-researcher` | Дослідження, аналіз, написання звітів. |

### 5. Контракт викликів інструментів (на що можна покластися)

- Проксі видає **рівно один `tool_calls` на крок** незалежно від стратегії — панель і суддя працюють цілком на боці сервера і ніколи не бачать вашу реальну схему `tools`, тож ваш наявний цикл інструментів працює без змін, а побічні ефекти лишаються рівно-один-раз.
- Коли останній результат інструмента виглядає як збій (error / exception / ненульовий код виходу), `fusion-agents` ескалує цей хід прямо в повний fusion — просити глибших міркувань самостійно не потрібно.
- Виклик інструмента, обрізаний лімітом токенів, повідомляється чесно (`stop_reason: "max_tokens"` на діалекті Anthropic), а не як нібито придатний до виконання поламаний JSON.

### 6. Готові скрипти запуску (замість ручного під'єднання)

```bash
./bin/fusion-opencode <модель>    # підніме проксі за потреби, запише конфігурацію провайдера OpenCode, відкриє OpenCode
./bin/fusion-claude <модель>      # підніме проксі за потреби, експортує змінні ANTHROPIC_*, відкриє Claude Code
```

Обидва враховують `FUSION_PROXY_URL` (типово `http://127.0.0.1:8080`) і `FUSION_PROXY_TOKEN`. Обидва приймають `run "<запит>"` після імені моделі для разового запуску без інтерфейсу.

### 7. Мінімальна перевірка (скопіюйте і запустіть)

```bash
curl -s http://127.0.0.1:8080/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"fusion-agents","messages":[{"role":"user","content":"Reply with the single word: ok"}]}'
```

JSON-відповідь означає, що все налаштовано. Детальне під'єднання Claude Code — у [`docs/claude-code.md`](./docs/claude-code.md); кожен ключ конфігурації задокументовано у [`fusion.example.yaml`](./fusion.example.yaml).
